## 1. 파이프라인 전체 구조와 데이터셋 매핑

```
┌─────────────────────────────────────────────────────────────┐
│              3단계 파이프라인 전체 구조                        │
├──────────────┬──────────────┬──────────────────────────────┤
│   Stage 1    │   Stage 2    │          Stage 3             │
│ YOLOv11-det  │ YOLOv11-obb  │  Alphabetical VLM            │
│ 레이아웃 감지  │ 주석 위치 감지 │  +                           │
│              │              │  Numerical VLM               │
├──────────────┼──────────────┼──────────────────────────────┤
│  Dataset 1   │  Dataset 2   │       Dataset 3              │
│  1,000장     │  1,406장     │  ~15,000 image-text pairs    │
└──────────────┴──────────────┴──────────────────────────────┘
```

각 단계는 **독립적인 전용 데이터셋**으로 학습되며, 이전 단계의 출력이 다음 단계의 입력으로 연결되는 구조입니다.

## 2. Dataset 1 — 레이아웃 감지 데이터셋 (YOLOv11-det)

### 2-1. 기본 정보

| 항목 | 내용 |
|------|------|
| 사용 모델 | YOLOv11-det (Stage 1) |
| 도면 수 | **1,000장** |
| 파일 형식 | PDF 및 이미지 형식 혼용 |
| 주석 방식 | 바운딩 박스(Bounding Box) 수동 레이블링 |
| 데이터 분할 | Train 80% (800장) / Test 20% (200장) |

### 2-2. 클래스 구성 및 레이블 수

| 클래스 | 레이블 수 | 비율 | 설명 |
|--------|-----------|------|------|
| **View** | 3,498 | 약 68% | 도면의 정투영도·단면도 등 시각적 뷰 영역 |
| **Notes** | 1,127 | 약 22% | 일반 공차, 표면 처리 등 텍스트 주석 영역 |
| **Title Block** | 458 | 약 9% | 도면 번호, 재료, 설계자 등 메타정보 표 |
| **합계** | **5,083** | 100% | |

> **클래스별 특징**
> - View가 압도적으로 많은 이유: 하나의 도면에 정면도·측면도·평면도·단면도 등 여러 뷰가 포함되기 때문
> - Title Block은 도면당 1개이므로 상대적으로 적음
> - Notes는 도면에 따라 0~수십 개까지 편차가 큼

### 2-3. 데이터 특징

**도면 다양성**
- 단순 단일 부품 도면부터 복잡한 다중 뷰 조립 도면까지 포함
- 정투영도(Orthographic views), 단면도(Sectional views) 모두 포함
- 다양한 도면 레이아웃과 템플릿 형식 반영

**주석(Annotation) 방식**
- 각 도면의 View, Title Block, Notes 영역을 **축 정렬 바운딩 박스(Axis-Aligned Bounding Box)** 로 수동 레이블링
- 회전된 요소 없음 (Stage 1은 레이아웃 수준이므로 OBB 불필요)

**학습 목적**
- 도면 전체에서 **어디에 무엇이 있는지** 파악
- 하위 단계(Stage 2, 3)를 위한 관심 영역(ROI) 분리

### 2-4. 학습 결과 성능

| 클래스 | 감지 정확도 |
|--------|-------------|
| View | **0.96** |
| Title Block | **0.99** |
| Notes | **0.98** |

## 3. Dataset 2 — 주석 위치 추정 데이터셋 (YOLOv11-obb)

### 3-1. 기본 정보

| 항목 | 내용 |
|------|------|
| 사용 모델 | YOLOv11-obb (Stage 2) |
| 도면 수 | **1,406장** |
| 데이터 출처 | 온라인 리포지토리 + 내부 아카이브 혼합 |
| 주석 방식 | 방향성 바운딩 박스(OBB, Oriented Bounding Box) 수동 레이블링 |
| 데이터 분할 | Train 80% (1,124장) / Test 20% (282장) |
| 처리 영역 | Stage 1에서 감지된 **View 영역 내부**만 대상 |

### 3-2. 클래스 구성 및 레이블 수

| 클래스 | 레이블 수 | 비율 | 설명 |
|--------|-----------|------|------|
| **Measures** | 9,663 | 약 74% | 치수선, 반지름(R), 나사(Thread) 콜아웃 포함 |
| **GD&Ts** | 3,215 | 약 25% | 기하공차 프레임 전체 (datum, 공차값, 기호 포함) |
| **Surface Roughness** | 152 | 약 1% | 표면 거칠기 기호 (Ra 값 등) |
| **합계** | **13,030** | 100% | |

> **클래스별 특징**
> - Measures가 압도적으로 많은 이유: 실제 도면에서 치수 주석이 가장 빈번하게 사용됨
> - Surface Roughness가 극히 적은 이유: 모든 표면에 기호가 표기되지 않으며, 일반 공차로 대체되는 경우 많음
> - GD&T는 정밀 부품 도면에 주로 등장하며, 조립 도면에는 상대적으로 드묾

### 3-3. 데이터 특징

**OBB(방향성 바운딩 박스) 사용 이유**
```
일반 Bounding Box (AABB):
┌─────────────────┐
│  ↗ 45° 회전된  │  ← 빈 공간이 많이 포함됨
│  치수 주석      │     → 부정확한 패치 추출
└─────────────────┘

OBB (Oriented Bounding Box):
  ╱──────╲
 ╱ 45° 치수╲  ← 객체에 딱 맞게 회전된 박스
╲   주석   ╱     → 정확한 패치 추출
  ╲──────╱
```

- 도면 내 주석은 수평·수직뿐 아니라 **임의의 각도**로 배치됨
- OBB는 객체의 실제 방향에 맞춰 회전된 박스를 예측
- 회전된 치수선, 기울어진 GD&T 프레임도 정확히 포착 가능

**클래스 불균형 특징**
```
Measures     ████████████████████████████████████ 9,663
GD&Ts        ████████████ 3,215
Roughness    ▌ 152
```
- 실제 산업 도면의 분포를 그대로 반영한 자연스러운 불균형
- Surface Roughness의 부족이 감지 정확도(0.54) 저하의 주원인

**다양한 드래프팅 스타일 포함**
- 온라인 리포지토리 + 내부 아카이브를 혼합하여 다양한 주석 스타일 확보
- ISO, ASME 등 다양한 표준의 GD&T 프레임 형식 포함

### 3-4. 학습 결과 성능

| 클래스 | 감지 정확도 | 비고 |
|--------|-------------|------|
| Measures | **0.95** | 충분한 샘플로 높은 정확도 |
| GD&Ts | **0.97** | 가장 높은 정확도 |
| Surface Roughness | **0.54** | 샘플 부족으로 낮은 정확도 |

## 4. Dataset 3 — VLM 학습 데이터셋 (Alphabetical + Numerical VLM)

### 4-1. 기본 정보

| 항목 | 내용 |
|------|------|
| 사용 모델 | Alphabetical VLM (Zero-shot) + Numerical VLM (파인튜닝) |
| 총 샘플 수 | **~15,000 image-text pairs** |
| 데이터 출처 | Dataset 1 + Dataset 2의 모든 주석 영역에서 추출 |

| 품질 관리 | 모든 주석 수동 검증 완료 |
| 클래스 비율 | 실제 분포 반영 유지 |

### 4-2. 서브셋 구성

#### 서브셋 A — Alphabetical (텍스트 영역)

| 항목 | 내용 |
|------|------|
| 샘플 수 | **~1,585개** |
| 포함 카테고리 | Title Block, Notes |

| 사용 모델 | Alphabetical VLM (**Zero-shot**, 파인튜닝 미수행) |

**Title Block JSON 출력 예시**
```json
{
  "Project_ID": "P0003",
  "Drawing_No": "P0003-000.004",
  "Title": "Vertical Steam Engine with Reverse Gear",
  "Part_Name": "Miniature Steam Engine",
  "Material": "Cast Iron",
  "Mass": "0.276 kg",
```

```json
  "notes": [

  ]
```

#### 서브셋 B — Numerical (수치 영역)

"수치 영역(Numerical)"은 단순히 숫자만을 의미하지 않습니다.  
도면에서 **스펙(사양)을 담은 기호 블록 전체**를 파싱하는 영역으로, 세 가지 카테고리로 구성됩니다.

| 항목 | 내용 |
|------|------|
| 샘플 수 | **~13,000개** |
| 포함 카테고리 | GD&T, Measures, Surface Roughness |
| 입력 | OBB 감지로 추출된 이미지 패치 (회전 보정 포함) |
| 출력 | 수치·기호 사양을 담은 JSON |
| 사용 모델 | Numerical VLM (**파인튜닝 수행**) |
| 데이터 분할 | Train 70% / Val 20% / Test 10% |

**① GD&T JSON 출력 예시**

도면의 GD&T 프레임은 공차 기호·수치·데이텀의 세 요소로 구성되며, 모두 하나의 문자열로 인코딩됩니다.

```
┌────────────────────────┐
│  ⊕  │  ⌀0.02  │  A   │
└────────────────────────┘
   ↑        ↑        ↑
공차기호   수치값   데이텀
```

```json
{ "GD&T": "Position ⌀0.02 A" }
```

> **읽는 법:** "기준면 A를 기준으로, 이 형상의 위치는 직경 0.02mm 원통 내에 있어야 한다"

| 구성 요소 | 값 | 의미 |
|-----------|-----|------|
| 공차 종류 | `Position` | 위치도 공차 |
| 공차 값 | `⌀0.02` | 허용 오차 범위 (직경 0.02mm) |
| 데이텀 | `A` | 기준면/기준축 |

> 세 요소가 모두 있어야 **하나의 완전한 GD&T 사양**이 됩니다.

**② Measure JSON 출력 예시**

```json
{ "Measures": ["34", "60", "20", "15", "102"] }
```

> **읽는 법:** "이 뷰(View)에서 감지된 치수 숫자들: 34mm, 60mm, 20mm, 15mm, 102mm"

| 특징 | 설명 |
|------|------|
| 배열 형태 | 뷰 안의 치수가 여러 개이므로 리스트로 출력 |
| 단위 없음 | 도면 표준상 mm가 기본이므로 숫자만 추출 |
| 순서 무관 | 위치 좌표 없이 값만 수집 (위치는 OBB가 이미 담당) |

**③ Surface Roughness JSON 출력 예시**

```json
{ "Roughness": "Ra 1.6 μm" }
```

> **읽는 법:** "산술평균 거칠기(Ra) 기준으로, 표면 거칠기는 1.6 마이크로미터 이하"

| 구성 요소 | 값 | 의미 |
|-----------|-----|------|
| 파라미터 | `Ra` | 산술평균 거칠기 (가장 일반적) |
| 수치 | `1.6` | 허용 거칠기 값 |
| 단위 | `μm` | 마이크로미터 |

> Ra 외에도 `Rz`, `Rmax` 등이 등장할 수 있으므로, 파라미터 종류까지 문자열에 포함합니다.

#### 세 카테고리 출력 비교

```
OBB 감지로 패치 추출
        ↓
   카테고리 판별
   ┌─────┬──────────┬────────────┐
  GD&T  Measures  Roughness
   ↓       ↓           ↓
단일 문자열  숫자 리스트  단일 문자열
```

| 카테고리 | 출력 형태 | 특징 |
|----------|-----------|------|
| GD&T | 단일 문자열 | 공차 종류 + 수치 + 데이텀을 하나의 표현으로 통합 |
| Measures | 숫자 배열 | 뷰 안의 치수가 여러 개이므로 리스트로 출력, 단위 생략 |
| Surface Roughness | 단일 문자열 | Ra/Rz 파라미터 종류와 수치 및 단위를 포함 |

> **Measures만 유일하게 복수 값을 배열로 담는 점**이 핵심 차이입니다.  
> 치수의 위치 정보는 OBB가 이미 담당하므로, VLM은 값만 추출합니다.

### 4-3. GD&T JSON 설계 원칙 — 공차 종류를 포함하는 이유

GD&T JSON에 `"Position"`과 같은 공차 종류 텍스트를 명시하는 이유는 **수치만으로는 사양의 의미가 불완전**하기 때문입니다.

```json
{ "GD&T": "⌀0.02 A" }          ← Position 없음 (불완전)
{ "GD&T": "Position ⌀0.02 A" }  ← Position 있음 (완전)
```

같은 `⌀0.02 A`라도 공차 종류가 다르면 의미가 완전히 달라집니다.

| GD&T 기호 | 공차 종류 | `⌀0.02`의 의미 |
|-----------|-----------|----------------|
| `⊕` | Position (위치도) | 중심점이 ⌀0.02 원통 안에 있어야 함 |
| `○` | Circularity (진원도) | 단면이 0.02mm 두께 띠 안에 있어야 함 |
| `⌭` | Cylindricity (원통도) | 전체 면이 0.02mm 두께 띠 안에 있어야 함 |
| `∥` | Parallelism (평행도) | A면과 평행 방향으로 0.02mm 이내 |

**도면 비교 시 필요성**

```
도면 A: { "GD&T": "Position    ⌀0.02 A" }
도면 B: { "GD&T": "Circularity  0.02"   }
```

- 공차 종류를 생략하면 → 두 도면이 **같은 부품처럼 오판**될 수 있음
- 공차 종류까지 저장하면 → **서로 다른 사양임을 정확히 판별** 가능

> Position은 데이텀 참조(`A`)와 직경 기호(`⌀`)를 사용하지만, Circularity(진원도)는 데이텀 없이 단일 수치만 사용합니다. **공차 종류에 따라 함께 등장하는 요소 자체가 달라지므로**, 공차 종류 텍스트는 반드시 보존되어야 합니다.

### 4-4. Donut 모델의 인식 방식

#### End-to-End 픽셀 기반 학습

Numerical VLM으로 사용된 Donut은 **OCR 엔진 없이** 픽셀 패턴에서 직접 텍스트 토큰을 출력하는 End-to-End 모델입니다.

```
도면 이미지 패치              Donut 모델              JSON 출력
┌─────────────┐                  ↓           ┌────────────────────────────┐
│ ⊕ │⌀0.02│A │  →  (시각적 이해)  →  │ "GD&T": "Position ⌀0.02 A" │
└─────────────┘                              └────────────────────────────┘
```

파인튜닝 과정에서 "이 모양의 픽셀 패턴 = Position이라는 토큰"을 직접 학습합니다.

```
학습 데이터 예시:
  [⊕ 기호가 있는 이미지]  →  정답: "Position ⌀0.02 A"
  [○ 기호가 있는 이미지]  →  정답: "Circularity 0.02"
  [∥ 기호가 있는 이미지]  →  정답: "Parallelism 0.05 A"
```

#### Unicode/OCR 방식과의 비교

| 구분 | Unicode/OCR 방식 | Donut 방식 |
|------|-----------------|------------|
| 인식 단위 | 문자 코드포인트 | 픽셀 패턴 |
| `⊕` 처리 | `U+2295`로 읽은 뒤 별도 매핑 테이블 필요 | 학습으로 직접 "Position" 출력 |
| 기호 왜곡/회전 | 취약 | 상대적으로 강건 |
| 추가 후처리 | 필요 | 불필요 |

#### 기호 대신 의미 문자로 출력하는 이유

```
기호 출력:  { "GD&T": "⊕ ⌀0.02 A" }          ← 후처리 시스템이 ⊕ 의미를 또 해석해야 함
문자 출력:  { "GD&T": "Position ⌀0.02 A" }    ← 사람과 시스템 모두 바로 이해 가능
```

#### Donut 내부 학습 포맷

Donut은 JSON을 그대로 받는 것이 아니라 **토큰 시퀀스**로 변환하여 학습합니다.

```python
# gt_parse 형태로 변환
{
  "gt_parse": { "GD&T": "Position ⌀0.02 A" }
}

# Donut 내부에서는 이렇게 토큰화됨
"<s_GD&T>Position ⌀0.02 A</s_GD&T>"
```

### 4-5. 데이터 구성 특징

**이미지 패치 추출 방식**
```
원본 도면
    ↓ Stage 1 (YOLOv11-det)
Title Block / Notes / View 영역 크롭
    ↓ Stage 2 (YOLOv11-obb) — View 영역만
OBB 기반 주석 패치 추출 + 회전 보정
    ↓
image-text pair 구성
```

**JSON 정답 레이블 구성 기준**
- Title Block / Notes: 구조화된 텍스트 + JSON 필드 수동 작성
- GD&T: 공차값, 기준(datum), 기호를 Unicode로 정확히 인코딩
- Measures: 수치값 배열로 구성
- Surface Roughness: Ra/Rz 파라미터 정확히 기록

**레이블 작성 예시**

추출된 패치 이미지에 대해 작성하는 JSON 레이블의 매핑 관계는 다음과 같습니다.

| 패치 이미지 내용 | JSON 레이블 |
|------------------|-------------|
| `⊕ │⌀0.02│ A` | `{ "GD&T": "Position ⌀0.02 A" }` |
| `∥ │ 0.05 │ B` | `{ "GD&T": "Parallelism 0.05 B" }` |
| `▽▽▽ Ra1.6` | `{ "Roughness": "Ra 1.6 μm" }` |
| `34, 60, 20` | `{ "Measures": ["34", "60", "20"] }` |

**레이블링 전략 3가지**

실제 데이터셋 구성 시 세 가지 접근법이 적용될 수 있으며, 각 방식은 정확도와 효율성 간의 트레이드오프를 가집니다.

*옵션 1 — 순수 수작업 (가장 정확, 가장 느림)*
```
작업자가 패치 이미지를 보고
   ↓
Label Studio / CVAT 같은 툴에서 JSON 직접 입력
   ↓
검수자 1명이 재확인
```

*옵션 2 — 반자동 (현실적 추천)*
```
1단계: PaddleOCR로 텍스트 초안 자동 추출
          ↓
2단계: 작업자가 오류 수정 (특수기호 ⊕ → Position 등)
          ↓
3단계: 검수
```

*옵션 3 — HITL 누적 방식 (반복 학습)*
```
초기 모델 예측값을 초안으로 사용
          ↓
작업자가 틀린 것만 수정
          ↓
수정된 데이터를 다시 학습에 추가
          ↓
반복 → 점점 수정량 감소
```

| 전략 | 정확도 | 속도 | 추천 단계 |
|------|--------|------|-----------|
| 순수 수작업 | ★★★ | ★ | 초기 시드 데이터 구축 |
| 반자동 (OCR 초안) | ★★ | ★★ | 대량 데이터 1차 확보 |
| HITL 누적 | ★★★ | ★★★ | 운영 단계 지속 개선 |

**레이블링 시 까다로운 부분**

| 문제 | 예시 | 해결 |
|------|------|------|
| 특수기호 매핑 | `⊕` → "Position" | 기호-텍스트 매핑표를 레이블 가이드에 포함 |
| 복합 공차 프레임 | 프레임이 2줄짜리인 경우 | 레이블 가이드에 표기 규칙 명시 |
| 이미지 품질 | 스캔본 흐림, 회전 | OBB 회전 보정 후 패치 추출 |
| 카테고리 혼동 | Measures vs GD&T 경계 | OBB 감지 단계에서 카테고리 선분리 |

**품질 관리**
- 모든 15,000개 샘플 수동 검증
- 실제 분포를 반영한 클래스 비율 유지
- Unicode 엔지니어링 기호 정확성 검증 (⌀, ±, ⊕ 등)

### 4-8. Numerical VLM 핵심 요약 (Q&A)

Section 4의 핵심 설계 의사결정을 질문 형태로 정리합니다.

| 질문 | 답변 |
|------|------|
| Numerical 영역이 왜 문자를 포함하나? | 숫자만으로는 사양의 의미가 불완전하기 때문 (`⌀0.02 A`는 어떤 공차인지 알 수 없음) |
| Position 문자를 왜 JSON에 넣나? | 같은 수치라도 공차 종류(Position·Flatness·Circularity 등)가 다르면 완전히 다른 사양이기 때문 |
| Donut은 어떻게 `⊕`를 Position으로 출력하나? | Unicode 코드포인트가 아닌 **픽셀 패턴 학습**으로 직접 토큰 출력 (OCR 엔진 없음) |
| 파인튜닝 데이터는 어떻게 만드나? | OBB 패치 추출 → 수작업/반자동 레이블링 → HITL 누적으로 점진적 개선 |
| Measures만 왜 배열인가? | 한 뷰에 다수의 치수가 존재하며, 위치 정보는 OBB가 이미 담당하므로 값만 모음 |

## 6. 데이터셋 한계 및 개선 방향

### 6-1. 클래스 불균형

| 클래스 | 샘플 수 | 문제 |
|--------|---------|------|
| Surface Roughness | 152 | 감지 정확도 0.54로 가장 낮음 |
| Title Block (VLM) | ~793 | 환각률 47.8%로 가장 높음 |

**개선 방향**
- 합성 데이터 증강(Synthetic Augmentation)으로 Surface Roughness 샘플 확보
- 데이터 오버샘플링 기법 적용

### 6-2. 도메인 편향

- 특정 산업(조선, 항공, 반도체 등) 도면 미포함
- 비표준 주석 형식 미반영
- 다국어 도면(한국어, 일본어 등 비라틴 문자) 미포함

**개선 방향**
- 다양한 산업군 도면 수집
- 다국어 Title Block 데이터 추가

### 6-3. 대형 조립 도면 부재

- 현재 데이터셋은 단순~중간 복잡도 부품 도면 중심
- 수백 개의 주석이 중첩된 대형 조립 도면 미포함

**개선 방향**
- 대형 조립 도면 수집 및 레이블링
- 다중 페이지 도면 처리 전략 개발

### 6-4. Alphabetical VLM 파인튜닝 데이터 부족

- ~1,585개 샘플로 파인튜닝 시 과적합(overfitting) 위험
- Title Block 스키마 다양성 부족

**개선 방향**
- Title Block 샘플 최소 5,000개 이상으로 확장
- 조직별 다양한 Title Block 템플릿 수집
