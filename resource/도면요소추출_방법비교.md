## 0. 한 줄 결론  *(0-A에서 논문·스코프 반영해 갱신됨)*

- **표제란(5필드 고정: Title/Drawing No./LIC. Material/Material/Rev)**: **Qwen zero-shot + ROI 전처리**로 이미 해결(현 구현). 추가 모델 불필요.
- **측정/Radii/GD&T**: **YOLO(det→obb) → fine-tuned Donut**이 문헌상 SOTA(GD&T F1 0.965, 경량 143M, 환각 낮음).
- **콜드스타트(실데이터 2장)**: 대형 VLM(Qwen/PaddleOCR) zero-shot으로 **자동 라벨링 → Donut 학습데이터 축적**.
- 핵심 원리: **"검출로 영역 분리 → 가능하면 fine-tuned 경량 Donut으로 끝낸다. 대형 VLM은 데이터 없을 때의 초기 학습데이터 마련(라벨 작성기)/보조."**

## 0-A. ★업데이트: 참고 논문 2건 반영 + 표제란 5필드 한정

> 이 절은 첨부 논문 2건과 **"표제란 = 5필드 고정"** 스코프를 반영한 **최신 결론**이며,
> 아래 섹션 3~7의 일부 평가(특히 Donut의 GD&T·표제란)를 **갱신**한다.

### (1) 참고 논문 (동일 연구진, 본 과제와 같은 주제)
- **[P1] From Drawings to Decisions** (Khan et al., 46p): YOLOv11-obb → fine-tuned 경량 VLM.
  1,367 도면 / 9범주 / 11,469 패치. **Donut F1 93.5%** (P 88.5 / R 99.2, 환각 11.5%)로 Florence-2보다 우수. Donut-base ≈ **143M**.
- **[P2] Multi-Stage Hybrid Framework** (Khan et al., 10p): YOLOv11-**det**(레이아웃: 뷰/표제란/노트) → **obb**(주석) → Donut 2종.
  **Numerical(치수·GD&T·거칠기) F1 0.963** (GD&T **0.965**, 거칠기 1.0, 치수 0.923),
  Alphabetical(표제란·노트, *미파인튜닝*) F1 0.672.

### (2) 논문이 바꾸는 2가지 평가
1. **GD&T는 fine-tuned Donut의 강점**(F1 0.965). 기존 "Donut GD&T=✕/○"는 *zero-shot·미학습* 한정. 유한 기호집합(14개)+스키마 제약이라 파인튜닝이 잘 수렴 → **◎로 상향**.
2. **fine-tuned 경량 Donut이 대형 zero-shot VLM(GPT-4o/Claude/Qwen)을 능가**(P1: Florence-2를 400장만 파인튜닝해도 GPT-4o·Claude-3.5 zero-shot 압도). 대형 zero-shot은 환각 잦음 → **production 정답은 fine-tuned Donut**, 대형 VLM은 초기 학습데이터 마련(라벨 작성기)/보조로 재배치.

### (3) 표제란 = 5필드 고정이라는 점이 중요
- 논문의 표제란 저조치(0.533)는 **표 전체 개방형 파싱 + 미파인튜닝** 결과 → **우리 5필드 과제엔 무관**.
- 5필드 난점은 "표 구조"가 아니라 ① 작은 글씨 읽기 ② 빈칸 판별(LIC) ③ Rev 전용 크롭 → **현재 Qwen zero-shot + ROI 전처리가 이미 해결**(`COMPARISON_RESULTS.md`: 환각 3건 교정).
- 5필드 난이도: Title(개방어휘 ★★★) > Drawing No./LIC/Rev(준정형 ★★) > Material(★).
- 결론: **표제란 5필드는 추가 모델/파인튜닝 불필요**. 대량화로 비용이 문제될 때만 5필드 전용 소형 Donut(스키마 좁아 라벨 적게 듦) 고려.

### (4) 정정 매트릭스 (핵심 행)
| 요소 | Donut(미파튜) | **Donut(파튜)** | PaddleOCR-VL | Qwen zero-shot |
|---|---|---|---|---|
| 표제란 5필드(고정) | △ | ○~◎ | ○(매핑 배관 필요) | **◎ 현 구현·검증** |
| 치수/Measures | ✕ | **◎ (F1 0.923)** | ◎ 저비용 | ○ |
| Radii/모따기 | ✕ | **◎** | ◎ | ○ |
| GD&T/데이텀 | ✕ | **◎ (F1 0.965)** | ✕(후처리) | ○(환각 위험) |
| 노트(자유문) | ✕ | △~○ | ○ | ◎ |

### (5) 재설정된 아키텍처 & 우선순위
```
[표제란 5필드]    → Qwen zero-shot + ROI 전처리        (현 구현, 유지)
[측정/Radii/GD&T] → YOLOv11-det(레이아웃) → YOLOv11-obb(주석) → fine-tuned Donut(통합)
                    (논문 SOTA: 경량 143M, GD&T 0.965, 환각 낮음)
```
- **1순위(데이터 확보 가능)**: 주석=fine-tuned Donut, 표제란 5필드=Qwen zero-shot.
- **콜드스타트(2장)**: Qwen/PaddleOCR zero-shot으로 자동 라벨링 → Donut 학습데이터 축적 → 파인튜닝.

### (6) 사소 정정
- Donut-base 파라미터: ~200M → **~143M**.
- vocab: 유한 GD&T 기호(14개)는 **Unicode로 직접 출력하도록 파인튜닝**해도 잘 됨(P1 Table 2). "영문 명칭 우회"는 선택지일 뿐, "vocab 부재"는 *zero-shot 한정* 문제.
- **검출을 2단계로 나누면 다중뷰 도면에서 정확도가 오른다.**
  한 페이지에 정면도·단면도·등각도 등 여러 뷰가 있어, 전체를 한 번에 검출하면 옆 뷰의 치수·기호가
  섞여 오검출된다. 그래서 ① **YOLOv11-det**로 페이지를 큰 영역(뷰 / 표제란 / 노트)으로 **먼저 분리**한 뒤,
  ② 잘라낸 **각 뷰 안에서만 YOLOv11-obb**로 주석(measure/gdt/radii)을 검출한다.
  (서랍별로 나눠 한 칸씩 뒤지는 셈 → 뷰 간 간섭↓, 글자도 상대적으로 커져 판독↑.)

> ⚠️ **읽기 안내**: 아래 §3~9는 *논문을 반영하기 전*에 쓴 초기 분석(대형 VLM 중심)이라
> **배경·참고용**이다. 본 **0-A가 논문을 반영한 최신 결론**이므로, §3~9와 0-A의 내용이 다르면
> **0-A를 따른다**. (옛 분석도 참고 가치가 있어 지우지 않고 남겨둠)

## 1. 대상 도면 특성 (실측)

검토에 사용한 실제 도면 2장은 모두 **텍스트 레이어가 없는 스캔본**이라 OCR/VLM이 필수다.

| 특성 | A1464068392 (A14-640083-9) | BH311726385 (BH3-117263-B) |
|---|---|---|
| 포맷 | A4 스캔(CCITT 흑백), **텍스트 0자** | A4 스캔(흑백+gray), **텍스트 ≈0자** |
| 뷰 | 정면도 + 단면 A-A | 정면도 + 단면 + 등각 + DETAIL 확대 |
| 치수 | Ø565, Ø409, Ø515, 16×Ø27, 11.25°, 모따기 24 | PCD110, Ø145/168, 6×홀, 22.2/19.5, 각도 다수 |
| Radii | R max 3 | R5.0 등 |
| 모따기/각도 | 24, 11.25° | 1×45°, 7×45°, 30°, 45° (밀집) |
| GD&T/데이텀 | 경미 | 데이텀 A 박스, 8×44.5 |
| 노트 | 적음 | 영문 NOTE 2줄 + 손글씨 한글 |
| 표제란 | 우하단 정형 표, 한/영 + 리비전 매트릭스 | 동일 형식, 한/영 혼용 |
| 노이즈 | **반투명 워터마크 오버레이** | 비교적 깨끗 |

**핵심 관찰 3가지**

1. 둘 다 **순수 스캔 → 텍스트 추출 불가 → OCR/VLM 필수**.
2. **워터마크·스캔 노이즈·작은 글씨** → 전처리(검출→크롭→확대)와 모델 견고성이 정확도를 좌우.
3. 물량 구성이 **전사(transcription) 성격 ≈ 80%**(치수·Ø·각도·모따기·표제란·노트),
   **의미 해석 ≈ 20%**(GD&T·데이텀) — *대략적 추정치*. → 비용 효율이 중요한 작업.

|---|---|---|
| 치수 / Ø / 각도 | 전사 | 작은 글씨, 워터마크 겹침, 다중 뷰 분산 |

## 3. 세 방법 비교

### 공통 전제: 왜 "YOLO +"인가
도면은 다중 뷰에 요소가 흩어져 있어 전체 페이지를 그대로 모델에 넣으면 정확도가 낮다.
**YOLO(-OBB)로 요소 위치를 먼저 검출 → 크롭·회전보정·확대 → 모델이 글자 읽기**.
"위치는 검출기/코드가, 글자는 모델이" 담당하는 역할 분리가 환각을 줄인다.

### 방법 1 — YOLO + fine-tuned Donut

- **성격**: 이미지→JSON을 **파인튜닝으로 통째 학습**하는 OCR-free 모델(Swin-B + BART, ~143M).
- **강점**: 표제란·단순치수처럼 **고정 스키마·반복 양식**에 탁월. 가볍고 빠르고 결정적·저비용.
- **약점**
  - **데이터 헌거**: 스키마당 보통 수백~수천 장 필요(원조는 실문서 1,100만 장(IIT-CDIP) +
    합성(SynthDoG)으로 사전학습).
  - **vocab 부재**: GD&T 글리프(`⊕ ⊥ ▱ ⌖ Ⓜ`)가 토크나이저에 거의 없음(단 `Ø`·`±`는 흔한
    문자라 예외). 토큰 추가는 가능하나 새 토큰은 랜덤 임베딩 → 안정화에 많은 예시 필요.
    희귀 클래스(gdt/radii)에서 특히 취약. **단, 출력을 영문 명칭으로 설계하면 우회 가능**(↓섹션 5 보충).
  - **추론 없음**: `⊕→position` 같은 해석을 일반화 못 하고 **본 조합만 재현**.
  - 스캔 노이즈/워터마크에 과적합형이라 취약.

### 방법 2 — YOLO + PaddleOCR-VL (0.9B)

- **성격**: **문서 파싱(OCR) 특화** VLM(NaViT 인코더 + ERNIE-4.5-0.3B). 출력은
  Markdown / HTML 표 / LaTeX 수식 같은 **전사 결과**(지시 기반 의미추출이 목적은 아님).
- **강점**
  - 표 인식 SOTA(OmniDocBench v1.5 종합 92.56) + **109개 언어(한국어 포함)**
    → 한/영 혼용 표제란·리비전 매트릭스·영문 노트 전사에 강함.
  - **0.9B 초경량** → 물량 많은 치수·Ø·R·각도 전사를 7B 대비 약 8배 가벼워 훨씬 싸고 빠르게.
  - 수식 head로 `±`, 분수형 공차, `Ø` 일부 유리.
- **약점**
  - **의미 구조화 없음** → 전사 결과를 별도 파서(규칙/소형 LLM)로 JSON화해야 함.
  - **GD&T niche 글리프**는 학습 분포 밖(단, 본 도면들은 GD&T 경미해 타격 작음).
  - 워터마크 견고성은 Qwen보다 검증 필요.

### 방법 3 — YOLO + fine-tuned Qwen2.5-VL-7B

- **성격**: **범용 추론 VLM**. 강한 문서/OCR 사전학습 + 지시 따르기 + 세계지식.
- **강점**
  - **스캔/워터마크 견고성 최고** ("워터마크 무시하고 치수만 읽기"가 프롬프트로 가능).
  - **소데이터 적합**: LoRA로 크롭 수백 장이면 스키마·기호·한영 혼용 동시 학습. 2장에서도 즉시 파일럿.
  - **GD&T 의미 해석**(characteristic/modifier/datum)을 단일 모델로 JSON까지.
- **약점**: **비용/속도/메모리**. 물량 많은 치수까지 7B로 돌리면 비효율.

## 4. 요소 × 방법 적합성 매트릭스 (본 도면 기준)

| 요소 | Donut | PaddleOCR-VL | Qwen-VL-7B |
|---|---|---|---|
| 치수/Ø/각도 (물량多) | ✕ | ◎ 저비용 | ○ (정확하나 과비용) |
| Radii / 모따기 | ✕ | ◎ | ○ |
| 표제란(한영 표) | △ | ◎ | ○ |
| 노트(영문+손글씨) | ✕ | ○ | ◎ |
| GD&T/데이텀(의미) | ✕ | ✕(후처리 필요) | ◎ |
| 워터마크/노이즈 견고성 | ✕ | △ | ◎ |
| 소데이터(2장 출발) | ✕ | ◎(zero-shot) | ◎(zero/few-shot) |
| 비용/속도 | ◎ | ◎ | △ |

범례: ◎ 매우 적합 / ○ 적합 / △ 제한적 / ✕ 부적합
(주: Qwen 소데이터 칸은 LoRA가 아니라 **프롬프트/few-shot** 기준 — 9-0 참조)

> ⚠️ **위 Donut 열의 전제**: 공차·치수·GD&T에 대해 **충분히 파인튜닝되지 않은(또는 학습 데이터가
> 적은) 상태**를 가정한 평가다. 즉 "지금 당장(실데이터 2장)"의 Donut 모습이다.
> Donut을 **제대로 파인튜닝하면** 평가가 크게 달라진다 → 아래 4-2.

<a id="s42"></a>

### 4-2. 만약 Donut을 충분히 파인튜닝한다면 (YOLO → fine-tuned Donut)

전제: **요소별 라벨 크롭을 충분히 확보**(실데이터 + 합성 데이터 수백~수천 장), `Ø/±/⊕` 등
필요한 토큰을 vocab에 추가 후 대량 학습, 스캔 노이즈 augmentation 적용.

| 요소 | 미파인튜닝(=섹션4) | **충분히 파인튜닝 시** | 비고 |
|---|---|---|---|
| 치수/Ø/각도 | ✕ | **◎** | 패턴 단순·반복 → 학습 쉽고 싸고 빠름 |
| Radii / 모따기 | ✕ | **◎** | 동일 |
| 표제란(한영 표) | △ | **○~◎** | Donut 본령(CORD형). **전체표 기준**(5필드면 0-A 참조). 텍스트·개방어휘라 숫자보다는 까다로움 |
| 노트(영문+손글씨) | ✕ | **△~○** | 자유문은 가변길이라 덜 친화, **손글씨 한글은 여전히 약함** |
| GD&T/데이텀(의미) | ✕ | **◎ (논문 F1 0.965)** | ★상향. 유한 기호집합+스키마 제약이라 파인튜닝이 잘 수렴(0-A 참조). 단 극희귀 조합은 augmentation 필요 |
| 워터마크/노이즈 견고성 | ✕ | **○** | augmentation으로 학습한 노이즈엔 강함, 새 노이즈엔 약함 |
| 소데이터(2장 출발) | ✕ | **✕ (그대로)** | ★ 본질적 한계 — 파인튜닝 자체가 데이터를 요구 |
| 비용/속도 | ◎ | **◎** | 최대 강점 유지(~143M, 7B 대비 수십 배 저렴·고속) |

**핵심 정리 (파인튜닝하면 무엇이 바뀌나)** — *논문 실측으로 보강(0-A)*
- 대부분의 `✕`는 **데이터·vocab 문제**여서 → 충분한 학습으로 **◎/○ 로 전환**된다.
  논문 실측: 전체 **F1 93.5%**, GD&T **0.965**, 거칠기 1.0, 치수 0.923.
  특히 **GD&T·치수 등 "숫자/기호" 범주가 ◎**(스키마 제약 덕에 가장 잘 수렴).
- 그러나 **두 가지는 파인튜닝해도 Qwen보다 약하다**:
  ① **GD&T 미학습 조합의 일반화**(추론이 없어 본 패턴만 재현),
  ② **자유문 노트·손글씨 한글**(가변·비정형).
- **변하지 않는 한 가지**: `소데이터(2장 출발)`의 `✕`. 파인튜닝은 **데이터가 전제**이므로,
  "지금 당장 적은 데이터로" 쓰려면 여전히 부적합 → 이때는 Qwen(LoRA)/PaddleOCR가 유리.

> 한 줄: **데이터(특히 합성)를 갖추면 Donut은 치수·GD&T·표제란에서 최강 가성비**가 되고
> (논문 GD&T F1 0.965), **노트(및 GD&T 미학습 조합)만** Qwen 보강이 필요하다.
> 결국 "**데이터 확보 여부**"가 Donut 채택의 갈림길.

## 5. Donut 파인튜닝 심화 — "괜찮지 않나?"에 대한 답

> 📌 본 절의 "GD&T가 Donut에 불리"는 **저데이터 한정** 서술이다. **충분히 파인튜닝하면
> GD&T는 오히려 Donut의 강점**(0-A: 논문 F1 0.965)임을 함께 보라.

**부분적으로 맞다.** Donut은 원래 이미지→JSON 파인튜닝 모델이므로:

- **표제란 / 단순치수**: 고정 스키마·반복 양식 → **Donut 파인튜닝이 정말 좋다**(싸고 빠르고 결정적).
- **GD&T / 공차**: 아래 3가지로 **데이터가 적을 때 다른 두 방법에 밀린다(dominated)**.
  1. vocab 부재(희귀 글리프),
  2. 추론 없음(조합마다 라벨 필요),
  3. 데이터 헌거(수백~수천 장 vs 현재 2장).

> **🔎 vocab 보충 — "유니코드에 있다 ≠ 모델이 잘 한다"**
> GD&T 기호 상당수는 유니코드 코드포인트가 있다(예: `⊥`U+22A5, `⌖`U+2316, `▱`U+25B1, `Ⓜ`U+24C2).
> 그러나 세 층위가 다르다: ① **유니코드 정의**(있음) → ② **토크나이저 vocab 포함**(모델마다 다름,
> Donut엔 거의 없음) → ③ **학습 노출**(희귀=잘 못 배움). byte-fallback로 *표현*은 되지만
> *인식·생성 신뢰성*은 ②③이 좌우한다.
> **두 가지 실천 포인트**:
> - **디코더(출력)**: 글리프 대신 **영문 명칭**(`<s_characteristic>position</s_characteristic>`)을 내보내면
>   흔한 단어만 생성 → vocab 문제 자체를 우회(토큰 추가·대량학습 불필요).
> - **인코더(이미지 이해)**: 유니코드와 무관. 글리프 **모양을 예시로 충분히 학습**해야 구분 가능.

### Donut을 진짜 살리는 길: 합성 데이터
GD&T/치수 크롭은 **프로그램으로 정답 JSON과 함께 대량 렌더링**할 수 있다(원조 Donut의 SynthDoG 방식).
→ vocab·데이터 헌거 동시 해결 → **YOLO + Donut(합성 대량 + 실데이터 소량)** 가 현실적 대안이 됨.
- 조건: ① 충실한 GD&T 렌더러 제작 공수, ② 합성→실스캔 도메인 갭을 augmentation + 실데이터로 보정.

## 6. 권장 아키텍처

논문(P1·P2) + 표제란 5필드 스코프를 반영한 **권장 구성**. 표제란 5필드는 Qwen zero-shot(현 구현),
주석은 fine-tuned Donut으로 끝낸다.

> 📌 **이 그림은 "목표(end-state)"다.** 현재는 검출기·Donut 학습 데이터가 없으므로 **콜드스타트
> (아래)로 데이터를 먼저 확보**해야 이 구성이 성립한다. 박스 끝의 상태 태그:
> `[현 구현]` 바로 동작 / `[학습 필요]` 데이터 확보·학습 후 / `[선택]` 상황 따라.

```
스캔 PDF → 300dpi 래스터화 → (워터마크 제거/이진화 = [선택], 실측 후)
   │
   ├─ 표제란(우하단) → 고정 ROI 크롭 (양식 고정이라 검출 불필요)
   │      → Qwen2.5-VL zero-shot + ROI 전처리 → 5필드 JSON      [현 구현]
   │
   ├─ 주석 검출: YOLOv11-det (뷰 분리) → 각 뷰 안에서 YOLOv11-obb
   │   │            (다중뷰 간섭↓, P2. 단순 도면이면 det 생략·obb 단독 가능)   [학습 필요]
   │   └─ measure / radii / gdt(+모따기는 measure에 포함) → deskew 크롭
   │          → fine-tuned Donut(통합) → 요소별 JSON           [학습 필요]
   │            (논문 SOTA: ~143M, GD&T 0.965)
   │
   └─ 노트 영역 → Qwen zero-shot 또는 PaddleOCR-VL 전사          [선택]
   │
   ▼ 병합 → { title_block(5필드), measures[](모따기 포함), radii[], gdt[], notes[] }
```

- **표제란**: 양식이 1~2종 고정이라 **고정 ROI**가 검출기보다 싸고 정확(현 구현도 det 없이 동작).
- **YOLOv11-det**: 표제란용이 아니라 **다중뷰 분리(주석 검출 전용)**. 뷰가 적은 도면이면 생략하고
  obb 단독(현 `train_obb.py` 구성)도 가능.
- **이진화**: Qwen/Donut은 RGB 입력이라 과한 이진화는 회색조 손실로 해로울 수 있음 → 실측 후 선택.
- **모따기(45° 등)**: 별도 클래스가 아니라 `measure` 로 포착(현 `drawing_obb.yaml` 3-클래스 유지).
  (단, §4·4-2 매트릭스에선 "단순 치수성"이 Radii와 같아 평가상 *"Radii/모따기"*로 묶어 표기 — 출력 위치는 measure.)

**콜드스타트(현재 단계, 학습데이터 없음)**: 측정/GD&T 크롭을 Qwen/PaddleOCR zero-shot으로 자동
라벨링(라벨 작성기) → Donut 학습데이터 축적 → 검출기·Donut 파인튜닝 후 위 "목표" 구성으로 전환.
(전사 vs 의미해석 비중은 대략 80:20 추정)

## 7. 상황별 결정 규칙

논문(0-A) 반영 기준. **정확도 최우선·production = fine-tuned Donut**, 대형 zero-shot은
콜드스타트 단계의 초기 학습데이터 마련(라벨 작성기)/보조다.

| 상황 | 권장 |
|---|---|
| 지금(실데이터 적음, 빠른 결과) | PaddleOCR-VL/Qwen zero-shot (초기 학습데이터 마련 = 자동 라벨링) |
| 표제란 **5필드** | **Qwen zero-shot + ROI**(현 구현) — 추가 모델 불필요(0-A) |
| 표제란 **전체표** 대량 자동화 | Donut 파인튜닝 (싸고 빠름) |
| 측정/Radii/GD&T (데이터 확보) | **fine-tuned Donut**(논문 SOTA, GD&T 0.965) |
| 정확도 최우선·production | **fine-tuned Donut**(대형 zero-shot은 환각으로 보조) |

## 8. 검증 제안 (실측 우선)

도면 표본이 적어 사전판단의 불확실성이 있으므로 소규모 파일럿 권장.

1. PDF를 300dpi PNG로 변환 → 요소별 크롭 30~50개 수작업 추출.
2. **동일 크롭**에 PaddleOCR-VL vs Qwen-VL 전사 정확도 비교
   (특히 워터마크 겹친 치수, 데이텀/노트).
3. 표제란은 PaddleOCR-VL 표 출력 품질을, GD&T는 Qwen 의미 JSON 정확도를 leaf-match로 측정.
4. 결과에 따라 "PaddleOCR 단독 / 하이브리드 분기점" 확정.
5. **(최신 경로) Donut 검증**: 2~3의 자동라벨 품질이 쓸 만하면 → 크롭을 Qwen으로 자동라벨(`donut_data`)
   → 검수 후 Donut 파인튜닝 → **held-out 실크롭**에서 leaf-match로 측정해 fine-tuned Donut이
   대형 zero-shot을 따라잡는지 확인(논문 기대치: GD&T F1 ≈ 0.965).

## 9. (부록) GD&T 크롭용 few-shot 프롬프트 초안

**콜드스타트 자동라벨(§10) 또는 zero-shot 운영**에서 GD&T 크롭을 Qwen-VL로 읽을 때 쓰는 프롬프트.
(최종 운영은 fine-tuned Donut이 맡지만, 그 **학습데이터를 만들려면 Qwen이 GD&T를 읽어 라벨을
생성**해야 한다 → 이 프롬프트가 그 도구.) **파인튜닝 없이** zero/few-shot으로 시작하고, 부족할 때만
LoRA로 넘어간다. 구성은 ① 텍스트 지시(기호→의미 매핑 + 스키마 + 규칙),
② few-shot 예시(이미지+정답 JSON), ③ Qwen 메시지 조립 코드.

<a id="s90"></a>

### 9-0. Qwen에 LoRA가 꼭 필요한가? — 아니오 (단계적 접근)

GD&T 크롭을 Qwen으로 읽을 때(§10 콜드스타트 자동라벨, 또는 zero-shot 운영) **LoRA는 선택**이다.
Qwen은 사전학습이 강해 **파인튜닝 없이도** 동작한다(이 프로젝트의 표제란 추출도 zero-shot으로
작동 중). **위에서 안 되면 아래로** 내려가는 단계적 접근을 권장:

| 단계 | 학습 | 방법 | 언제 충분한가 |
|---|---|---|---|
| 1. Zero-shot | 0 | 스키마 + 규칙 + 기호 매핑표 프롬프트 | 흔한 기호, 소량 처리 |
| 2. **Few-shot** | 0 | 예시 크롭 2~5개 in-context (아래 ①~③) | 형식 흔들림·정확도 보강 → **대부분 여기서 해결** |
| 3. LoRA | 있음 | 라벨 크롭 수백 장 학습 | 아래 "필요 조건"이 남을 때만 |

**LoRA가 실제로 필요한 경우만**
- 특정 niche 글리프를 **반복적으로** 오인할 때
- **대량 처리** + 스키마 일관성/환각 억제가 중요할 때
- 유럽식 쉼표 등 도메인 관습을 강제하고 싶을 때

**비용 관점**: LoRA는 라벨 수백 장 + GPU 학습 + 유지보수가 든다. → **먼저 프롬프트(1·2단계)로
정확도를 실측하고, 목표 미달인 "구체적 실패 유형"이 남을 때만 그 유형을 겨냥해 학습**한다.
GD&T 양이 적은 본 두 도면이라면 **1·2단계만으로 충분할 가능성이 높다.**

### ① 텍스트 프롬프트 (instruction)

```text
You are reading a single CROPPED region from a mechanical engineering drawing.
The crop contains ONE GD&T (Geometric Dimensioning & Tolerancing) feature control
frame, possibly with leading datum target text. Transcribe it and return STRICT JSON.

## GD&T symbol → characteristic (use the English name in "characteristic")
  ⏤  straightness          ⊥  perpendicularity
  ⏥ / ▱  flatness          ∠  angularity
  ○  circularity           ∥  parallelism
  ⌭  cylindricity          ⌖ / ⊕  position
  ⌒  profile of a line     ◎  concentricity
  ⌓  profile of a surface  ⌯  symmetry
  ↗  circular runout       ⌰  total runout

## Material-condition modifiers (the circled letter)
  Ⓜ = "M" (MMC)   Ⓛ = "L" (LMC)   Ⓢ = "S" (RFS)   Ⓟ = "P" (projected zone)

## Output JSON schema (these keys exactly)
{
  "characteristic": <english name from the table, or null>,
  "symbol":         <the glyph as printed, e.g. "⌖", "⊥", "▱">,
  "tolerance":      <tolerance zone value as printed, e.g. "Ø0.4", "0.08">,
  "modifier":       <"M" | "L" | "S" | "P" | null  — modifier on the tolerance value>,
  "datums":         <ordered array of datum refs WITH modifiers, e.g. ["A","B(M)","C"]; [] if none>,
  "raw":            <verbatim transcription of the whole frame, left to right>
}

## Rules
- Copy text VERBATIM. Preserve European decimal commas ("0,1" stays "0,1"; never "0.1").
- "Ø" means a diameter tolerance zone — keep it in "tolerance".
- A circled letter AFTER the tolerance value → "modifier". A circled letter attached to a
  datum letter → keep it inside that datum entry, e.g. B with Ⓜ becomes "B(M)".
- Datums keep left-to-right order as printed.
- If a field is not present, use null (or [] for "datums").
- Output ONLY the JSON object. No markdown fences, no explanation.
```

### ② Few-shot 예시 (이미지 ↔ 정답 JSON)

샘플 도면 `doc_sample_paper_01.jpg`의 실제 패치 3개 사용. **예시는 텍스트만이 아니라
이미지 크롭 자체를 함께 넣어야** in-context 학습이 된다.

| 예시 크롭(이미지) | 기대 출력 JSON |
|---|---|
| `⊕ Ø0.4Ⓜ A BⓂ C` | `{"characteristic":"position","symbol":"⌖","tolerance":"Ø0.4","modifier":"M","datums":["A","B(M)","C"],"raw":"⊕ Ø0.4Ⓜ A BⓂ C"}` |
| `⊥ 0.1Ⓜ A` | `{"characteristic":"perpendicularity","symbol":"⊥","tolerance":"0.1","modifier":"M","datums":["A"],"raw":"⊥ 0.1Ⓜ A"}` |
| `▱ 0.08` | `{"characteristic":"flatness","symbol":"▱","tolerance":"0.08","modifier":null,"datums":[],"raw":"▱ 0.08"}` |

### ③ Qwen2.5-VL 메시지 조립 코드

```python
import json
from transformers import AutoProcessor, Qwen2_5_VLForConditionalGeneration
from qwen_vl_utils import process_vision_info

MODEL_ID = "Qwen/Qwen2.5-VL-7B-Instruct"
INSTRUCTION = "...①의 텍스트 프롬프트 전체..."

# few-shot 예시: (이미지경로, 정답 JSON) — 실제 크롭 파일로 교체
SHOTS = [
    ("examples/gdt_position.png",
     {"characteristic":"position","symbol":"⌖","tolerance":"Ø0.4","modifier":"M",
      "datums":["A","B(M)","C"],"raw":"⊕ Ø0.4Ⓜ A BⓂ C"}),
    ("examples/gdt_perp.png",
     {"characteristic":"perpendicularity","symbol":"⊥","tolerance":"0.1","modifier":"M",
      "datums":["A"],"raw":"⊥ 0.1Ⓜ A"}),
    ("examples/gdt_flatness.png",
     {"characteristic":"flatness","symbol":"▱","tolerance":"0.08","modifier":None,
      "datums":[],"raw":"▱ 0.08"}),
]

def img(path):
    return {"type": "image", "image": f"file://{path}"}

def build_messages(query_crop_path):
    messages = [{"role": "user", "content": [{"type": "text", "text": INSTRUCTION}]}]
    for path, gold in SHOTS:                         # 예시: 이미지→user, 정답→assistant
        messages.append({"role": "user",
                         "content": [img(path), {"type": "text", "text": "Extract:"}]})
        messages.append({"role": "assistant",
                         "content": [{"type": "text",
                                      "text": json.dumps(gold, ensure_ascii=False)}]})
    messages.append({"role": "user",                 # 실제 질의 크롭
                     "content": [img(query_crop_path), {"type": "text", "text": "Extract:"}]})
    return messages

model = Qwen2_5_VLForConditionalGeneration.from_pretrained(
    MODEL_ID, torch_dtype="auto", device_map="auto")
processor = AutoProcessor.from_pretrained(MODEL_ID)

messages = build_messages("crops/some_gdt_crop.png")
text = processor.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
image_inputs, _ = process_vision_info(messages)
inputs = processor(text=[text], images=image_inputs, padding=True,
                   return_tensors="pt").to(model.device)
out = model.generate(**inputs, max_new_tokens=256, do_sample=False)   # greedy=결정적
trimmed = [o[len(i):] for i, o in zip(inputs.input_ids, out)]
raw = processor.batch_decode(trimmed, skip_special_tokens=True)[0].strip()
result = json.loads(raw)   # 기존 parse_json()으로 감싸 안전 파싱 권장
```

### 사용 팁
- 예시 이미지 3장은 **실제 크롭 파일**(`examples/*.png`)로 교체해야 효과가 난다(텍스트 표만으론 부족).
- 정확도 부족 시 → 예시를 2→4~5개로 늘리고, 자주 틀리는 특성(runout 계열 등)을 우선 포함.
- `parse_json()`(기존 `extract_title_block.py`)으로 감싸 코드펜스/잡텍스트를 방어.
- 그래도 특정 글리프가 반복 오인되면 → 그때 비로소 LoRA.

## 10. (부록) 콜드스타트: 대형 VLM 자동라벨 → Donut 학습데이터 축적

0-A(5)·섹션 6의 **"콜드스타트 변형"** 을 자세히 풀어 쓴다. 핵심은 **비싼·똑똑한 모델의 지능을
싸고 빠른 모델로 옮겨 담는** 일회성 초기 학습데이터 마련(라벨 작성기; 지식 증류 distillation).

### (1) 풀려는 문제 — "닭과 달걀"
fine-tuned Donut을 쓰려면 **이미지+정답 JSON 쌍 수백~수천 개**가 필요하다. 그런데
정답을 사람이 일일이 타이핑하면 느리고, 데이터가 없으면 Donut을 시작조차 못 한다(교착).

### (2) 핵심 아이디어 — "선생이 정답지를 써주고 학생이 공부"
| 모델 | 성격 |
|---|---|
| 대형 VLM(Qwen/PaddleOCR) | 느리고 비싸지만 **학습 없이(zero-shot)** 바로 JSON 생성 |
| Donut(~143M) | 빠르고 싸지만 **학습 데이터가 있어야** 동작 |

→ **대형 VLM이 정답지(라벨)를 자동 작성 → 그걸로 Donut을 학습.** 학습을 마친 Donut은
선생(대형 VLM) 없이도 혼자 빠르게 처리한다.

### (3) 단계 흐름 (`donut_data` 파이프라인)
```
① 크롭 모으기   도면→YOLO 검출/크롭(또는 수작업)        →  patches/<class>/*.png
② 자동 라벨링   각 크롭을 Qwen zero-shot 입력            →  {"characteristic":"position", ...}
   (build_dataset)  (= 대형 VLM이 정답지 초안 작성)
③ 저장/축적     이미지+JSON 쌍 + manifest.csv           →  datasets/donut_anno/
④ 사람 검수     manifest 보고 틀린 것만 교정(verified=1) ←  ★ 품질 관문
⑤ 변환          to_donut_vml.py                         →  donut_vml 학습 포맷 + CFG
⑥ Donut 학습    donut_vml 파인튜닝                       →  production용 fine-tuned Donut
```
②가 "자동 라벨링", ③ 누적이 "학습데이터 축적"이다.

### (4) 예시 (GD&T 크롭 1장)
```
크롭:  [ ⊕ Ø0.4Ⓜ A B C ]   ← 이미지만, 정답 없음
② Qwen zero-shot 출력(정답지 초안):
   {"characteristic":"position","tolerance":"Ø0.4","modifier":"M",
    "datums":["A","B","C"],"raw":"⊕ Ø0.4Ⓜ A B C"}
③ 이 (이미지 ↔ JSON) 쌍 = Donut 학습 1건
```

### (5) ★ 가장 중요한 주의 — 자동 라벨은 "초안"이다
대형 VLM도 틀린다(예: `Drawing No.` 숫자 1자 오인). **틀린 정답지로 배우면 학생도 틀린다
(garbage in → garbage out).** 그래서:
- `manifest.csv` 의 **`verified` 열**로 사람이 빠르게 훑어 **틀린 것만 교정**(0에서 타이핑 X).
- 사람은 **라벨러가 아니라 검수자** → 노력 대폭 절감. `parse_ok=0`(파싱 실패)부터 우선 확인.

### (6) 왜 이렇게 하나 — 경제성
- Qwen으로 운영을 계속 돌리면 도면마다 7B 호출 → 느리고 GPU·비용 큼.
- **Qwen은 라벨링에 "한 번만" 쓰고, 운영은 싸고 빠른 Donut**(~143M)으로 → 대량처리에서 수십 배 절감.

### (7) 선순환(flywheel)
```
Qwen 자동라벨 → Donut 학습 → Donut이 새 도면 처리 → 결과 재검수·축적
                                   → 데이터↑ → Donut 재학습 → 더 정확 → Qwen 의존↓ ...
```

> 한 줄: **데이터가 없을 때 똑똑한 Qwen을 "정답지 작성기"로 한 번 써서 이미지↔JSON을 자동
> 생성하고(사람은 검수만), 그걸로 싸고 빠른 Donut을 길러 실전 투입**하는 초기 학습데이터 마련(라벨 작성기) 전략.

## 11. (부록) vocab 문제는 "파인튜닝하면 사라진다"

0-A(6)의 *"유한 GD&T 기호(14개)는 Unicode로 직접 출력하도록 파인튜닝해도 잘 됨. 영문 명칭
우회는 선택지일 뿐, 'vocab 부재'는 zero-shot 한정 문제"* 를 자세히 푼다.

### (1) vocab 문제의 3층 (복습)
| 층위 | 의미 | zero-shot Donut |
|---|---|---|
| ① 유니코드 정의 | 문자가 표준에 존재 | ✅ (⊕ ⊥ ▱ …) |
| ② 토크나이저 vocab 포함 | 모델이 토큰으로 가짐 | ❌ 거의 없음 |
| ③ 학습 노출 | 그 토큰을 충분히 학습 | ❌ 못 배움 |

zero-shot에선 ②③이 안 돼 GD&T 기호를 못 읽/못 냄 = **"vocab 부재"**.

### (2) 핵심: GD&T 기호는 14개로 **유한**
ASME Y14.5의 표준 기호 14개로 고정(위치 ⊕, 직각도 ⊥, 평면도 ▱ …). **유한하므로** 토큰 추가도
예시 확보도 현실적이다 — 이게 파인튜닝으로 해결되는 근거.

### (3) 파인튜닝이 ②③을 직접 해결
- ② 14개 글리프를 vocab에 **추가**(임베딩 테이블 확장),
- ③ 그 기호가 든 크롭을 **수백 장 학습** → 비어 있던 임베딩이 의미를 얻음.
→ zero-shot에서 "없던/못 배운" 토큰이 **정상 학습 토큰**이 된다. **P1은 14개를 Unicode로
인코딩(Table 2)해 글리프를 직접 출력하도록 파인튜닝 → F1 0.965**로 잘 작동.

### (4) 출력 설계는 두 선택지, 둘 다 OK(파인튜닝 전제)
| 설계 | 예 | 비고 |
|---|---|---|
| (A) Unicode 직접 | `<s_symbol>⊕</s_symbol>` | 14개 토큰 추가+학습 (P1 방식) |
| (B) 영문 명칭 | `<s_characteristic>position</s_characteristic>` | 흔한 단어라 토큰 추가도 불필요 |

(B)가 학습 부담이 약간 적은 정도이지 (A)도 충분. **"영문 명칭 우회"는 필수가 아니라 선택.**

### (5) 결론: "vocab 부재"는 zero-shot 한정
- **zero-shot**: 토큰 추가·학습 불가 → 사전학습이 본 것에 갇힘 → 희귀 글리프 실패 = vocab이 실제 문제.
- **fine-tuning**: vocab을 직접 통제(추가)하고 예시로 학습 → **문제 자체가 소멸**.

> 한 줄: GD&T 기호는 14개로 유한 → 파인튜닝 때 토큰 추가·학습하면 Unicode 글리프 직접 출력도
> 잘 된다. "영문 명칭 변환"은 선택, "vocab에 없다"는 걱정은 **학습을 안 하는 zero-shot에서만** 유효.

## 12. (부록) 환각은 파인튜닝으로 해결되나

질문: *"Qwen을 공차 등으로 파인튜닝하면 환각이 해결되나?"* → **크게 줄지만 완전히 사라지진 않는다.**
그리고 이 작업의 환각 1차 방어선은 *파인튜닝이 아니라 전처리·구조*다.

### (1) 파인튜닝이 줄여주는 부분 ✅
엄격한 JSON 스키마 + 도메인 데이터로 파인튜닝하면:
- **형식 흔들림↓**(엉뚱한 키·구조 억제), **도메인 관습 학습**(유럽식 쉼표·`Ø`·기호, `Rev`↔`Security` 혼동 교정)
- **빈칸→옆값 복사**도 *빈칸=null 예시*를 학습에 넣으면 감소

### (2) 그래도 "해결"은 아니다 ⚠️
- VLM은 확률적 생성 → 잔여 환각 상존. **논문(P1)도 파인튜닝한 Donut 환각률 11.5%**(0 아님).
- **진짜 안 보이는 입력**(블러·워터마크·잘림)엔 "모르면 비워라"를 명시 학습하지 않으면 그럴듯한 값을 지어냄.
- 학습에 없던 새 양식/조합엔 여전히 취약.

### (3) 더 중요: 이 프로젝트의 환각은 *전처리*가 먼저 잡았다
`COMPARISON_RESULTS.md`의 환각 3건 교정은 **파인튜닝이 아니라 ROI 크롭·확대 + Rev/LIC 전용 읽기**였다.
→ **1차 방어 = 검출→크롭→작고 깨끗한 입력 + 스키마/null 규율**, 파인튜닝은 그 위 2차 보강.

### (4) 비용 — Qwen 파인튜닝은 비싼 해법
Qwen(7B) LoRA로 환각을 줄일 수 있으나 학습·서빙이 무겁다. **같은 낮은 환각을 논문은 fine-tuned
Donut(~143M)으로 더 싸게** 달성(GD&T F1 0.965). → 권장은 "Qwen 파인튜닝"보다 "(주석은) Donut 파인튜닝".

### (5) 환각을 실제로 낮추는 조합
| 수단 | 효과 |
|---|---|
| 검출→크롭→ROI 확대 | 작고 깨끗한 입력 → 1차 차단 (현 구현) |
| strict JSON + **null 강제(abstention)** | "모르면 비움" → 지어내기↓ |
| greedy 디코딩 | 결정적·무작위성 제거 |
| 도메인 파인튜닝(Donut 권장 / Qwen 선택) | 형식·관습·필드혼동 환각↓ |
| 검증 가드레일(스키마 검사·저신뢰 사람 검수) | 잔여 환각 포착(`manifest verified`) |

> 한 줄: 파인튜닝은 환각을 **크게 줄이지만 완전 해결은 아니며**, 이 작업에선 **전처리(ROI)+스키마/null
> 규율이 먼저**이고 같은 효과를 **fine-tuned Donut이 더 저렴하게** 낸다.
