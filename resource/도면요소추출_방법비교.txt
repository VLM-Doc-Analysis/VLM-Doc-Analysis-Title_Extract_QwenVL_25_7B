====================================================================
 도면 요소 -> JSON 추출: 방법 비교 분석
====================================================================
대상  : 기계 도면(플랜지)에서 공차/치수/GD&T/Radii/모따기/표제란/노트를
        인식하여 구조화(JSON)로 추출
방법  : (1) YOLO + Donut
        (2) YOLO + PaddleOCR-VL(0.9B)
        (3) YOLO + Qwen2.5-VL-7B
작성  : 2026-06-15


--------------------------------------------------------------------
0. 한 줄 결론
--------------------------------------------------------------------
(0-A에서 논문·스코프 반영해 갱신됨)
- 표제란(5필드 고정: Title/Drawing No./LIC. Material/Material/Rev):
    Qwen zero-shot + ROI 전처리로 이미 해결(현 구현). 추가 모델 불필요.
- 측정/Radii/GD&T:
    YOLO(det->obb) -> fine-tuned Donut이 문헌상 SOTA(GD&T F1 0.965, 경량 143M, 환각 낮음)
- 콜드스타트(실데이터 2장):
    대형 VLM(Qwen/PaddleOCR) zero-shot으로 자동 라벨링 -> Donut 학습데이터 축적
- 핵심 원리:
    "검출로 영역 분리 -> 가능하면 fine-tuned 경량 Donut으로 끝낸다.
     대형 VLM은 데이터 없을 때의 부트스트랩/보조."


--------------------------------------------------------------------
0-A. [업데이트] 참고 논문 2건 반영 + 표제란 5필드 한정
--------------------------------------------------------------------
이 절은 첨부 논문 2건과 "표제란=5필드 고정" 스코프를 반영한 최신 결론이며,
아래 섹션 3~7의 일부 평가(특히 Donut의 GD&T/표제란)를 갱신한다.

(1) 참고 논문 (동일 연구진, 본 과제와 같은 주제)
  [P1] From Drawings to Decisions (Khan et al., 46p):
       YOLOv11-obb -> fine-tuned 경량 VLM. 1,367 도면/9범주/11,469 패치.
       Donut F1 93.5% (P 88.5 / R 99.2, 환각 11.5%), Florence-2보다 우수.
       Donut-base ~= 143M.
  [P2] Multi-Stage Hybrid Framework (Khan et al., 10p, arXiv 2510.21862v2):
       YOLOv11-det(레이아웃) -> obb(주석) -> Donut 2종.
       Numerical(치수/GD&T/거칠기) F1 0.963 (GD&T 0.965, 거칠기 1.0, 치수 0.923),
       Alphabetical(표제란/노트, *미파인튜닝*) F1 0.672.

(2) 논문이 바꾸는 2가지 평가
  1) GD&T는 fine-tuned Donut의 강점(F1 0.965). 기존 "Donut GD&T=✕/○"는
     zero-shot/미학습 한정. 유한 기호집합(14개)+스키마 제약 -> 파인튜닝 잘 수렴 -> ◎ 상향.
  2) fine-tuned 경량 Donut이 대형 zero-shot VLM(GPT-4o/Claude/Qwen)을 능가
     (P1: Florence-2 400장 파인튜닝이 GPT-4o/Claude-3.5 zero-shot 압도).
     대형 zero-shot은 환각 잦음 -> production 정답은 fine-tuned Donut, 대형은 보조.

(3) 표제란 = 5필드 고정이 중요
  - 논문의 표제란 저조치(0.533)는 "표 전체 개방형 파싱 + 미파인튜닝" 결과 -> 5필드엔 무관.
  - 5필드 난점은 표 구조가 아니라 1)작은글씨 2)빈칸판별(LIC) 3)Rev 전용 크롭
    -> 현재 Qwen zero-shot + ROI 전처리가 이미 해결(COMPARISON_RESULTS: 환각 3건 교정).
  - 난이도: Title(개방어휘 ★★★) > Drawing No./LIC/Rev(★★) > Material(★).
  - 결론: 표제란 5필드는 추가 모델/파인튜닝 불필요. 대량화 시에만 5필드 소형 Donut 고려.

(4) 정정 매트릭스 (핵심 행)
  요소                  Donut(미파튜)  Donut(파튜)        PaddleOCR-VL    Qwen zero-shot
  --------------------  ------------  ----------------  --------------  ---------------
  표제란 5필드(고정)    △             ○~◎              ○(매핑 배관)    ◎ 현구현·검증
  치수/Measures         ✕             ◎ (F1 0.923)      ◎ 저비용        ○
  Radii/모따기          ✕             ◎                 ◎              ○
  GD&T/데이텀           ✕             ◎ (F1 0.965)      ✕(후처리)       ○(환각 위험)
  노트(자유문)          ✕             △~○               ○              ◎

(5) 재설정된 아키텍처 & 우선순위
  [표제란 5필드]    -> Qwen zero-shot + ROI 전처리      (현 구현, 유지)
  [측정/Radii/GD&T] -> YOLOv11-det(레이아웃) -> YOLOv11-obb(주석) -> fine-tuned Donut(통합)
                       (논문 SOTA: 경량 143M, GD&T 0.965, 환각 낮음)
  - 1순위(데이터 확보 가능): 주석=fine-tuned Donut, 표제란 5필드=Qwen zero-shot
  - 콜드스타트(2장): Qwen/PaddleOCR zero-shot 자동 라벨링 -> Donut 학습데이터 축적 -> 파인튜닝

(6) 사소 정정
  - Donut-base 파라미터: ~200M -> ~143M
  - vocab: 유한 GD&T 기호(14개)는 Unicode로 직접 출력하도록 파인튜닝해도 잘 됨(P1 Table 2).
    "영문 명칭 우회"는 선택지. "vocab 부재"는 zero-shot 한정 문제.
  - 검출을 2단계(det 레이아웃 -> obb 주석)로 두면 다중뷰 간섭 감소.

[!] 아래 섹션 3~9는 초기 분석(대형 VLM 중심)이며, 상충 시 본 0-A가 우선한다.

[A1464068392 / A14-640083-9]
  - A4 스캔(CCITT 흑백), 텍스트 0자
  - 뷰: 정면도 + 단면 A-A
  - 치수: Ø565, Ø409, Ø515, 16xØ27, 11.25도, 모따기 24

  - GD&T/데이텀: 경미
  - 표제란: 우하단 정형 표, 한/영 + 리비전 매트릭스
  - 노이즈: 반투명 워터마크 오버레이 (★OCR 난점)

[BH311726385 / BH3-117263-B]
  - A4 스캔(흑백+gray), 텍스트 거의 0자
  - 뷰: 정면도 + 단면 + 등각 + DETAIL 확대
  - 치수: PCD110, Ø145/168, 6x홀, 22.2/19.5, 각도 다수
  - Radii: R5.0 등
  - 모따기/각도: 1x45, 7x45, 30, 45 (밀집)
  - GD&T/데이텀: 데이텀 A 박스, 8x44.5
  - 노트: 영문 NOTE 2줄 + 손글씨 한글
  - 표제란: 동일 형식, 한/영 혼용

핵심 관찰 3가지
  1) 둘 다 순수 스캔 -> 텍스트 추출 불가 -> OCR/VLM 필수
  2) 워터마크/스캔 노이즈/작은 글씨 -> 전처리(검출->크롭->확대)와
     모델 견고성이 정확도를 좌우
  3) 물량 구성: 전사 성격 약 80%(치수/Ø/각도/모따기/표제란/노트),
     의미 해석 약 20%(GD&T/데이텀) [대략적 추정치] -> 비용 효율이 중요

3. 세 방법 비교
--------------------------------------------------------------------
[공통 전제] 왜 "YOLO +" 인가
  도면은 다중 뷰에 요소가 흩어져 전체 페이지를 그대로 넣으면 정확도 낮음.
  YOLO(-OBB)로 위치 검출 -> 크롭/회전보정/확대 -> 모델이 글자 읽기.
  "위치는 검출기/코드, 글자는 모델" 역할 분리 -> 환각 감소.

(1) YOLO + Donut
  - 성격: 이미지->JSON을 파인튜닝으로 통째 학습(Swin-B+BART, ~143M), OCR-free
  - 강점: 표제란/단순치수 등 고정 스키마/반복 양식에 탁월.
          가볍고 빠르고 결정적, 저비용.
  - 약점:
      * 데이터 헌거: 스키마당 보통 수백~수천 장 필요
        (원조는 실문서 1,100만(IIT-CDIP) + 합성(SynthDoG) 사전학습)
      * vocab 부재: GD&T 글리프(⊕ ⊥ ▱ ⌖ Ⓜ)가 토크나이저에 거의 없음
        (단 Ø·±는 흔한 문자라 예외). 토큰 추가는 가능하나 새 토큰=랜덤 임베딩
        -> 많은 예시 필요. 단, 출력을 영문 명칭으로 설계하면 우회 가능(섹션5 보충).
      * 추론 없음: 기호->의미 일반화 불가, 본 조합만 재현
      * 스캔 노이즈/워터마크에 취약

(2) YOLO + PaddleOCR-VL (0.9B)
  - 성격: 문서 파싱(OCR) 특화 VLM(NaViT + ERNIE-4.5-0.3B).
          출력=Markdown/HTML표/LaTeX 전사 (의미추출이 목적 아님)
  - 강점:
      * 표 인식 SOTA + 109개 언어(한국어) -> 한/영 표제란/노트 전사 강함
      * 0.9B 초경량 -> 물량 많은 치수 전사를 7B 대비 약 8배 가벼워 훨씬 싸고 빠르게
      * 수식 head로 ±, 분수형 공차, Ø 일부 유리
  - 약점:
      * 의미 구조화 없음 -> 별도 파서(규칙/소형 LLM)로 JSON화 필요
      * GD&T niche 글리프는 학습 분포 밖 (단, 본 도면은 GD&T 경미)
      * 워터마크 견고성은 Qwen보다 검증 필요

(3) YOLO + Qwen2.5-VL-7B (fine-tuned)
  - 성격: 범용 추론 VLM(강한 문서/OCR 사전학습 + 지시 따르기 + 세계지식)
  - 강점:
      * 스캔/워터마크 견고성 최고("워터마크 무시하고 치수만" 프롬프트 가능)
      * 소데이터 적합: LoRA 크롭 수백 장이면 스키마/기호/한영 동시 학습
      * GD&T 의미 해석을 단일 모델로 JSON까지
  - 약점: 비용/속도/메모리. 물량 많은 치수까지 7B면 비효율

[!] 위 Donut 열의 전제:
    공차/치수/GD&T에 대해 "충분히 파인튜닝되지 않은(또는 데이터가 적은)"
    상태를 가정한 평가 = 지금 당장(실데이터 2장)의 Donut.
    Donut을 제대로 파인튜닝하면 평가가 크게 달라진다 -> 아래 4-2.


--------------------------------------------------------------------
4-2. 만약 Donut을 충분히 파인튜닝한다면 (YOLO -> fine-tuned Donut)
--------------------------------------------------------------------
전제: 요소별 라벨 크롭을 충분히 확보(실데이터+합성 수백~수천 장),
      Ø/±/⊕ 등 필요한 토큰을 vocab에 추가 후 대량 학습,
      스캔 노이즈 augmentation 적용.

요소                   미파인튜닝   충분히 파인튜닝 시   비고
---------------------  ----------  ------------------  ----------------------------
치수/Ø/각도            ✕           ◎                  패턴 단순 -> 학습 쉽고 싸고 빠름
Radii / 모따기         ✕           ◎                  동일
표제란(한영 표)        △           ◎                  Donut 본령(CORD형), 한글토큰 학습 시
노트(영문+손글씨)      ✕           △~○                자유문 덜 친화, 손글씨 한글 여전히 약함
GD&T/데이텀(의미)      ✕           ◎ (논문 F1 0.965)  ★상향. 유한기호집합+스키마제약 -> 잘 수렴(0-A)
워터마크/노이즈        ✕           ○                  학습한 노이즈엔 강함, 새 노이즈엔 약함
소데이터(2장 출발)     ✕           ✕ (그대로)         ★ 본질적 한계 - 파인튜닝은 데이터가 전제
비용/속도              ◎           ◎                  최대 강점 유지(~143M, 7B 대비 수십배 저렴)

핵심 정리 (파인튜닝하면 무엇이 바뀌나) - 논문 실측 보강(0-A)
  - 대부분의 ✕는 데이터/vocab 문제 -> 충분한 학습으로 ◎/○ 로 전환.
    논문 실측: 전체 F1 93.5%, GD&T 0.965, 거칠기 1.0, 치수 0.923.
    특히 GD&T/치수 등 숫자·기호 범주가 ◎ (스키마 제약 덕에 가장 잘 수렴).
  - 그러나 두 가지는 파인튜닝해도 Qwen보다 약함:
      (1) GD&T 미학습 조합의 일반화 (추론 없어 본 패턴만 재현)
      (2) 자유문 노트 / 손글씨 한글 (가변/비정형)
  - 변하지 않는 한 가지: '소데이터(2장 출발)'의 ✕.
    파인튜닝은 데이터가 전제 -> 지금 당장 적은 데이터로는 여전히 부적합
    -> 이때는 Qwen(LoRA)/PaddleOCR가 유리.

  한 줄: 데이터(특히 합성)를 갖추면 Donut은 치수/표제란에서 최강 가성비가 되고,
         GD&T/노트만 Qwen 보강이 필요. 결국 "데이터 확보 여부"가 갈림길.


--------------------------------------------------------------------
5. Donut 파인튜닝 심화 - "괜찮지 않나?"에 대한 답
--------------------------------------------------------------------
부분적으로 맞다. Donut은 원래 이미지->JSON 파인튜닝 모델이므로:

  - 표제란/단순치수: 고정 스키마/반복 양식 -> Donut 파인튜닝이 정말 좋음
    (싸고 빠르고 결정적)
  - GD&T/공차: 데이터가 적을 때 다른 두 방법에 밀림(dominated)
      1) vocab 부재(희귀 글리프)
      2) 추론 없음(조합마다 라벨 필요)
      3) 데이터 헌거(수백~수천 장 vs 현재 2장)

[vocab 보충: "유니코드에 있다 != 모델이 잘 한다"]
  GD&T 기호 상당수는 유니코드에 있음(⊥ U+22A5, ⌖ U+2316, ▱ U+25B1, Ⓜ U+24C2).
  그러나 세 층위가 다름:
    1) 유니코드 정의(있음)
    2) 토크나이저 vocab 포함(모델마다 다름, Donut엔 거의 없음)
    3) 학습 노출(희귀=잘 못 배움)
  byte-fallback로 표현은 되나 인식/생성 신뢰성은 2),3)이 좌우.
  실천 포인트:
    - 디코더(출력): 글리프 대신 영문 명칭(position 등) 출력 -> vocab 문제 우회
    - 인코더(이미지): 유니코드 무관, 글리프 모양을 예시로 충분히 학습해야 구분

[Donut을 진짜 살리는 길: 합성 데이터]
  GD&T/치수 크롭은 프로그램으로 정답 JSON과 함께 대량 렌더링 가능
  (원조 Donut의 SynthDoG 방식).
  -> vocab/데이터 헌거 동시 해결
  -> YOLO + Donut(합성 대량 + 실데이터 소량)이 현실적 대안
  조건: (1) 충실한 GD&T 렌더러 제작 공수
        (2) 합성->실스캔 도메인 갭을 augmentation + 실데이터로 보정

9. (부록) GD&T 크롭용 few-shot 프롬프트 초안
--------------------------------------------------------------------
하이브리드에서 GD&T 크롭을 Qwen-VL로 읽을 때 쓰는 프롬프트.
파인튜닝 없이 zero/few-shot으로 시작하고, 부족할 때만 LoRA로 넘어감.
구성: (1) 텍스트 지시 (2) few-shot 예시(이미지+JSON) (3) Qwen 메시지 조립 코드.

[9-0] Qwen에 LoRA가 꼭 필요한가? -> 아니오 (단계적 접근)
....................................................................
섹션 6의 "Qwen2.5-VL-7B (프롬프트 우선, 필요시 LoRA)"는 LoRA가 선택이라는 뜻.
Qwen은 사전학습이 강해 파인튜닝 없이도 동작(이 프로젝트 표제란도 zero-shot).
위에서 안 되면 아래로 내려가는 단계적 접근 권장:

  단계         학습   방법                              언제 충분한가
  -----------  -----  --------------------------------  -----------------------
  1.Zero-shot  0      스키마+규칙+기호매핑표 프롬프트    흔한 기호, 소량
  2.Few-shot   0      예시 크롭 2~5개 in-context         형식보강 -> 대부분 해결
  3.LoRA       있음   라벨 크롭 수백 장 학습            아래 조건 남을 때만

  LoRA가 실제로 필요한 경우만:
    - 특정 niche 글리프를 반복적으로 오인
    - 대량 처리 + 스키마 일관성/환각 억제가 중요
    - 유럽식 쉼표 등 도메인 관습 강제

  비용: LoRA는 라벨 수백 장 + GPU 학습 + 유지보수.
    -> 먼저 프롬프트(1·2단계)로 정확도 실측, 목표 미달 실패유형이 남을 때만
       그 유형 겨냥해 학습. GD&T 양 적은 본 도면은 1·2단계만으로 충분 가능성↑.

[1] 텍스트 프롬프트 (instruction)
....................................................................
You are reading a single CROPPED region from a mechanical engineering
drawing. The crop contains ONE GD&T feature control frame, possibly with
leading datum target text. Transcribe it and return STRICT JSON.

## GD&T symbol -> characteristic (use English name in "characteristic")
  straightness(⏤)  perpendicularity(⊥)  flatness(⏥/▱)  angularity(∠)
  circularity(○)   parallelism(∥)        cylindricity(⌭) position(⌖/⊕)
  profile-line(⌒)  profile-surface(⌓)    concentricity(◎) symmetry(⌯)
  circular-runout(↗)                      total-runout(⌰)

## Modifiers (circled letter)
  Ⓜ="M"(MMC)  Ⓛ="L"(LMC)  Ⓢ="S"(RFS)  Ⓟ="P"(projected zone)

## Output JSON schema (these keys exactly)
{
  "characteristic": <english name, or null>,
  "symbol":         <glyph as printed, e.g. "⌖","⊥","▱">,
  "tolerance":      <zone value as printed, e.g. "Ø0.4","0.08">,
  "modifier":       <"M"|"L"|"S"|"P"|null  -- modifier on tolerance value>,
  "datums":         <ordered refs WITH modifiers, e.g. ["A","B(M)","C"]; [] if none>,
  "raw":            <verbatim transcription, left to right>
}

## Rules
- Copy VERBATIM. Preserve European commas ("0,1" stays "0,1", never "0.1").
- "Ø" = diameter zone -> keep in "tolerance".
- circled letter AFTER tolerance -> "modifier";
  circled letter ON a datum -> inside that entry, e.g. B+Ⓜ = "B(M)".
- datums keep left-to-right order; missing field -> null (or [] for datums).
- Output ONLY the JSON object. No fences, no explanation.
