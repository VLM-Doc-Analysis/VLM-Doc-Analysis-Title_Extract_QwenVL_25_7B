# 스캔 도면 검출 — 데이터 준비 통합 가이드

> **기준 논문**: *A Multi-Stage Hybrid Framework for Automated Interpretation of Multi-View Engineering Drawings Using Vision Language Model* (Khan et al.)
>
> **대상 파이프라인 (3단계)**
> - **Stage 1**: 레이아웃 검출 (View, Title Block, Notes) — YOLOv11-**det**
> - **Stage 2**: 주석 검출 (Measures, GD&T, Surface Roughness) — YOLOv11-**obb**
> - **Stage 3**: 의미 파싱 — Donut 기반 VLM 2종
>
> **전제**: 입력은 **스캔된 PDF 도면** (1-bit 흑백·노이즈·기울임·압축 아티팩트)
>
> **목적**: 데이터 부족 상황에서 라벨링을 크게 늘리지 않고 검출 성능을 끌어올리기 위한 **소스 점검 → 해상도 → 전처리 → 증강** 전 과정 정리

# A. 전체 그림

## 1. 전체 데이터 준비 흐름

작업은 실제 데이터 흐름 순서대로 진행합니다. **소스 점검이 맨 앞**이라는 점이 핵심입니다.

```
[스캔 PDF]
   │
   ▼
① 소스 점검  (pdfimages -list)           ← §2  가장 먼저
   └─ 1-bit / grayscale / RGB · 원본 DPI 확인
   │
   ▼
② 이미지 추출 & 해상도 결정               ← §3~5
   ├─ 원본 그대로 추출 (재래스터화 지양)
   ├─ DPI: 300 하한 / 400 권장
   ├─ Stage 1: 전체 도면 → imgsz 1024
   └─ Stage 2: View crop → 1024 타일 (통짜 리사이즈 금지)
   │
   ▼
③ 전처리  (최소화, 학습·추론 공통)        ← §6
   ├─ DPI/해상도 정규화          ✅ 유지
   ├─ Deskew                     ⚠️ 큰 기울임만 / 작으면 증강으로
   ├─ 회색조 + CLAHE             ✅ 이진화 대신
   └─ 강한 이진화·denoise        ❌ 생략 권장
   │
   ▼
④ 데이터 증강  (학습 셋에만)              ← §7~11
   ├─ 기하학적 (약하게)
   ├─ 스캔 열화 (강하게) ★핵심
   ├─ 명암
   └─ 객체 단위 (copy-paste, mosaic)
   │
   ▼
⑤ 학습 (YOLOv11) → 진단 → ablation       ← §12~14
```

> **일관성 원칙**: ②③(추출·해상도·전처리)은 **학습·추론·증강 모두에 동일하게** 적용해야 합니다.
> 추론 때만 빠지면 증강·전처리 효과가 무너집니다.

# B. 소스 & 해상도

## 2. 소스 점검 — 가장 먼저

대부분 건너뛰지만 **가장 중요한 첫 단계**입니다. 스캔 PDF 내부 래스터의 비트 깊이와 해상도가 이후 모든 결정을 좌우합니다.

```bash
# poppler — 임베디드 이미지의 비트수·해상도·압축 확인
pdfimages -list drawing.pdf

# 원본 이미지를 재샘플링 없이 그대로 추출
pdfimages -all drawing.pdf out_prefix
```

### 비트 깊이별 분기

| 소스 형식 | 흔한가 | 의미 |
|-----------|--------|------|
| **1-bit (CCITT G4 / fax 압축)** | 🔵 매우 흔함 | **이미 흑백 2치로 굳어 들어옴** → 이진화 논쟁 무의미 |
| **8-bit grayscale** | 보통 | 회색조 유지 vs 이진화 선택이 의미 있음 (→ 회색조 권장) |
| **RGB** | 가끔 | 도장·승인란 색 등. 보통 회색조로 떨궈도 무방 |

> **왜 1-bit가 핵심인가**
> 도면은 선·글자뿐인 선화라 흑백 2색으로 저장하면 용량이 극단적으로 작아집니다.
> 그래서 도면 스캔은 관행적으로 **CCITT G4**(흑백 압축)로 저장됩니다 — 즉 **스캐너가 이미 이진화를 끝내 놓은** 상태입니다.
> 한번 흑백으로 굳으면 회색 음영은 되살릴 수 없으므로, 실제 고민은 "이진화를 추가할지"가 아니라
> **"이미 들어온 흑백을 어떻게 다룰지"** 입니다.

### 재래스터화보다 원본 추출

스캔 PDF를 임의 DPI로 다시 렌더링하면 원본과 안 맞아 **보간 아티팩트**가 생깁니다(원본 200DPI를 300으로 올리면 없는 디테일을 지어냄). 가능하면 `pdfimages -all`로 **원본 해상도 그대로 추출**해 쓰세요.

## 3. 해상도 설계 원리: 가장 작은 기호 기준

검출 모델은 객체가 입력에서 너무 작으면(**대략 20px 미만**) 급격히 못 잡습니다.
따라서 **가장 작은 의미 있는 클래스(Surface Roughness)가 입력에서 최소 ~32px**가 되도록 거꾸로 해상도·타일을 정합니다.

### DPI별 거칠기 기호 픽셀 크기 (물리적 3~5mm 가정)

| DPI | 3mm 기호 | 5mm 기호 | 판단 |
|-----|---------|---------|------|
| 150 | ~18px | ~30px | 🔴 부족 |
| **300** | ~35px | ~59px | 🟡 하한 |
| **400** | ~47px | ~79px | 🟢 여유 |

> **논문과의 연결 — Surface Roughness 0.54**
> 논문은 Stage 2의 Roughness 검출 정확도 0.54를 **데이터 불균형(152개)**으로만 설명합니다.
> 하지만 작은 기호는 **해상도·imgsz에도 크게 좌우**됩니다. 전체 View를 640으로 리사이즈했다면
> 50px 기호가 ~10px로 뭉개집니다. 즉 **불균형 + 해상도**가 함께 작용했을 가능성이 큽니다.
> (논문은 DPI·imgsz·타일링을 전혀 보고하지 않습니다 — 이 가이드가 그 빈칸을 채웁니다.)

---

## 4. Stage 1 해상도 — 전체 도면 (레이아웃)

타깃(View, Title Block, Notes)이 **큰 영역**이라 고해상도가 필요 없습니다.

- 전체 도면 래스터화 → **imgsz 1024~1280** 으로 리사이즈해서 학습/추론
- **타일링 불필요** (영역이 페이지의 1/4 ~ 전체 수준)
- **1024로 시작**, 멀티뷰가 매우 밀집하면 1280

> Stage 1은 해상도 승부처가 아닙니다. 가볍게 가고 연산은 Stage 2에 투자하세요.

---

## 5. Stage 2 해상도·타일링 — View crop ★핵심

해상도 설계의 승부처입니다. **큰 View crop을 통째로 640으로 줄이면 작은 기호가 소멸**합니다.

| 방식 | 50px 거칠기 기호의 입력 내 크기 | 결과 |
|------|------------------------------|------|
| 통짜 리사이즈 (3000px → 640) | ~10px | 🔴 소멸 |
| **1024 타일 (리사이즈 없음)** | ~50px | 🟢 검출 가능 |

### 방식 A — crop이 작을 때 (≤ ~1280px)
- View crop을 **imgsz 1024~1280** 으로 그대로 OBB에 투입 (타일링 불필요)

### 방식 B — crop이 클 때 (수천 px, 밀집) → 권장
- View crop을 **1024px 타일 + 15~25% 오버랩**으로 잘라 타일마다 OBB 추론 → 병합
- **핵심 장점**: 리사이즈로 줄이지 않으니 **기호의 원본 픽셀 크기가 그대로 보존**됨

> **⚠️ OBB 병합 주의**: OBB는 회전 박스라 타일 경계 병합에 **rotated NMS**가 필요합니다.
> SAHI를 쓴다면 **현재 버전의 OBB 지원 여부를 반드시 확인**하세요(축정렬 박스 위주라 커스텀 병합이 필요할 수 있음).

# C. 전처리

## 6. 전처리 — 어디까지 필요한가

흐름도(§1)의 전처리 항목이 **모두 필수는 아닙니다.** 딥러닝 검출은 고전 OCR과 달라, 변동을 "전처리로 없앨지 / 증강으로 견디게 할지"를 항목마다 따로 판단합니다.

### 6-1. 항목별 필요성

| 단계 | 정말 필수인가 | 판단 |
|------|--------------|------|
| **DPI/해상도 정규화** | 🟢 거의 필수 | 작은 기호 정보량을 좌우 → 유지 |
| **Deskew** | 🟡 조건부 | 작은 기울임은 회전 증강으로 대체 가능 |
| **이진화** | 🔴 불필요 (종종 해로움) | 보통 빼고 회색조 사용 |
| **노이즈 제거** | 🟡 약하게/생략 | 열화 증강과 모순될 수 있음 |

### 6-2. 왜 그런가

**DPI/해상도 정규화 — 유지.** 진짜 목적은 "DPI 맞추기"가 아니라 작은 기호가 충분한 픽셀을 갖게 하는 것(§3).

**Deskew — 조건부 & 도면에서는 주의.** 텍스트 문서의 전역 deskew(Hough/projection)는 지배적 텍스트 baseline 기준인데, **도면은 온갖 방향 선이 가득해 전역 deskew가 엉뚱하게 틀어버리는 사고가 잦습니다.** ① ±2° 이내면 회전 증강으로 흡수하고 생략, ② 꼭 보정하려면 **도면 외곽 테두리나 Title Block 사각형**을 기준으로.

**이진화 — 보통 제거. ⚠️** 공격적 이진화는 **얇은 선과 작은 기호를 지워버립니다** — 정확히 검출 대상을. CNN은 회색조를 잘 처리하고 정보를 더 보존합니다. → **이진화 대신 회색조 + 가벼운 대비 정규화(CLAHE)**. (단, §2에서 소스가 이미 1-bit면 이 선택지 자체가 없음.)

**노이즈 제거 — 약하게/생략.** 증강에서 노이즈를 일부러 더하는데(§8) 전처리에서 빼고 다시 더하는 건 모순. 가벼운 median despeckle 정도만.

### 6-3. 결정 원칙 — 정규화 vs 증강

- **전처리로 정규화** → 입력 분포가 좁아져 학습이 쉬워짐. 단, **추론 때도 동일 적용 필요**, 정보 손실 위험(특히 이진화).
- **증강으로 견디게** → 정보 보존, 모델이 변동 흡수. 단, 데이터·용량이 더 필요.

> **핵심**: 전처리 단계를 하나 추가할 때마다 학습-추론이 어긋날 지점이 하나 늘어납니다.
> **꼭 필요한 것만 남기는 것**이 유지보수·성능 양쪽에 유리합니다.

# D. 데이터 증강

## 7. 증강 ① 핵심 원리: 두 가지 목적

증강을 무작정 섞으면 효과를 분석할 수 없습니다. 도면 검출에서 증강은 목적이 둘로 나뉩니다.

| 목적 | 무엇을 하나 | 대표 기법 |
|------|------------|-----------|
| **① 일반화 (variation)** | 위치·크기·맥락 변화에 둔감해지게 | 회전, 스케일, copy-paste, mosaic |
| **② 도메인 정합 (domain matching)** | 학습 데이터를 **실제 스캔본 분포**에 맞춤 | 노이즈, 압축, 번짐, 기울임 |

> **스캔 PDF에서는 ② 도메인 정합이 핵심.** 깨끗한 도면으로만 학습하면 실제 스캔본에서 무너집니다.

---

## 8. 증강 ② 기법 4종 분류

### 8-1. 기하학적 증강 — **"약하게"**

도면은 방향·텍스트가 의미를 가지므로 사진처럼 막 변형하면 안 됩니다.

**✅ 안전 (권장 범위)**

| 기법 | 범위 | 이유 |
|------|------|------|
| 약한 회전 | ±2~5° | 스캔 기울어짐 모사 |
| 약한 스케일/줌 | 0.8~1.2배 | 크기 변화 일반화 |
| 약한 평행이동 | ±5% | 위치 변화 일반화 |
| 원근·탄성 왜곡 | 약하게 | 스캔 굴곡 모사 |
| 랜덤 크롭/타일링 | — | **소형 기호 비율↑ (매우 효과적)** |

**❌ 금지**

| 기법 | 이유 |
|------|------|
| 좌우/상하 반전 | GD&T 기호 방향성·텍스트 손상 |
| 90°/큰 각도 회전 | 의미 파괴 |
| 강한 전단(shear) | 기호 형태 왜곡 |

### 8-2. 스캔 열화 증강 — **"강하게" (스캔 PDF의 핵심)**

실제 스캔 결함을 코드로 재현한다고 생각하세요.

| 기법 | 모사 대상 |
|------|-----------|
| 가우시안/모션 블러 (약하게) | 스캔 초점 흐림 |
| 가우시안 + salt-and-pepper 노이즈 | 스캔 잡티 |
| **JPEG 압축 (quality 30~70)** | **스캔 PDF 강압축 — 실제 갭의 큰 부분** |
| **Dilation / Erosion (morphological)** | 잉크 번짐 / 선 끊김 |
| 종이 질감·얼룩·접힘·그림자 합성 | 스캔 배경 결함 |
| 다운샘플 후 업샘플 | 저해상도 스캔 (작은 기호 뭉개짐) |

> **핵심 팁**: 강도는 **실제 스캔본 품질 분포를 보고 맞추세요.** 과하게 망가뜨리면 도메인이 오히려 멀어집니다.

### 8-3. 명암 증강

흑백·회색조 도면이 많아 **색 변형은 거의 무의미**, 명암만 유효합니다.

| 기법 | 용도 |
|------|------|
| 밝기/대비/감마 (약하게) | 스캔 노출 차이 |
| CLAHE | 흐린 스캔 대비 개선 |
| ~~색조·채도~~ | 흑백 도면에선 생략 |

### 8-4. 객체 단위 증강 — **소형·희소 기호 해결의 열쇠**

| 기법 | 설명 | 주의점 |
|------|------|--------|
| **Copy-paste (실제 스캔 패치)** | 실제 도면에서 잘라낸 GD&T·거칠기·치수 패치를 빈 View 영역에 붙이고 bbox 새로 생성 | 경계는 살짝 블렌딩. 희소 클래스를 더 많이 붙여 불균형 완화 |
| **Mosaic** | 4장을 한 장으로 합쳐 다양한 맥락·스케일 학습 | **밀집 소형 객체에선 객체가 더 작아져 해로울 수 있음** → 약하게/후반 epoch엔 `close_mosaic` |
| **Cutout / Random Erasing** | 일부를 가려 occlusion에 강건 | 작은 기호를 통째로 지우지 않게 패치 크기 제한 |

> **왜 "실제 스캔 패치"인가**: 합성 기호와 실제 스캔 기호는 질감 차이가 큽니다.
> 배경(진짜 스캔 종이)+객체(진짜 스캔 기호) 둘 다 실제여야 도메인 갭이 최소화됩니다.
> → 특히 **Roughness 152개 불균형** 해결의 핵심 수단 (§12와 연계).

## 9. 증강 ③ 적용 방식 & bbox 동기화

### 9-1. 온라인 vs 오프라인

| 방식 | 설명 | 적합한 경우 |
|------|------|------------|
| **온라인 증강** (권장) | 학습 중 매 epoch마다 실시간 변형 | 가벼운 변형 전반 |
| **오프라인 증강** | 미리 변형본을 파일로 생성 | copy-paste 합성 등 무겁거나 검수가 필요한 작업 |

→ **두 방식을 섞는 것이 최선**: 무거운 합성은 오프라인, 가벼운 변형은 온라인.

### 9-2. 가장 중요한 원칙 — bbox 동기화

기하학적 변형을 하면 **bounding box 좌표도 똑같이 변형**돼야 합니다.

- **Albumentations** 같은 bbox-aware 라이브러리 사용 (직접 구현은 실수 잦음)
- 회전·크롭 후 box가 밖으로 나가면 자동 제거되도록 `min_visibility` 설정
- **학습 전 반드시 몇 장을 시각화**해서 box가 정확히 따라오는지 눈으로 확인

## 10. 증강 ④ 실전 Albumentations 파이프라인

스캔 도면 검출에 맞춘 온라인 증강 예시 (bbox 동기화 포함):

```python
import albumentations as A

transform = A.Compose([
    # --- ① 기하학적 (약하게) ---
    A.Affine(rotate=(-4, 4), scale=(0.85, 1.15),
             translate_percent=(-0.05, 0.05), p=0.7),
    A.RandomResizedCrop(size=(1024, 1024), scale=(0.6, 1.0), p=0.5),
    A.Perspective(scale=(0.02, 0.05), p=0.2),   # 스캔 굴곡

    # --- ② 스캔 열화 (핵심) ---
    A.OneOf([
        A.GaussianBlur(blur_limit=(3, 5)),
        A.MotionBlur(blur_limit=5),
    ], p=0.4),
    A.GaussNoise(var_limit=(10, 50), p=0.4),
    A.ImageCompression(quality_lower=30, quality_upper=70, p=0.5),  # 스캔 압축
    A.OneOf([
        A.Morphological(operation="dilation", p=1.0),  # 잉크 번짐
        A.Morphological(operation="erosion",  p=1.0),  # 선 끊김
    ], p=0.3),
    A.Downscale(scale_min=0.5, scale_max=0.9, p=0.2),  # 저해상도 스캔

    # --- ③ 명암 ---
    A.RandomBrightnessContrast(brightness_limit=0.2,
                               contrast_limit=0.2, p=0.4),
    A.CLAHE(clip_limit=2.0, p=0.2),
],
bbox_params=A.BboxParams(
    format="yolo",          # YOLO 라벨 포맷
    min_visibility=0.3,     # 30% 미만 보이면 box 제거
    label_fields=["class_labels"],
))
```

> **참고**: `Morphological`은 Albumentations 버전에 따라 이름이 다를 수 있습니다.
> 설치 버전을 확인하고, 없으면 OpenCV로 직접 dilation/erosion을 구현해 넣으면 됩니다.

## 11. 증강 ⑤ 적용 순서 & 강도 튜닝

### 11-1. 순서 — 실제 스캔 과정을 따라가기

```
기하학적  →  열화  →  압축
(기울어진 종이 → 스캐너가 흐릿하게 찍고 → PDF로 압축)
```

### 11-2. 강도 튜닝 절차

1. **약한 설정으로 시작** → baseline보다 좋아지는지 확인
2. **한 종류씩 강도를 올려가며** 검증 셋 성능 비교 (ablation)
3. 진단:
   - 학습 손실↓ 인데 검증 성능 정체 → 증강이 **너무 약함**
   - 학습이 불안정 → 증강이 **너무 셈**

# E. 학습 & 운영

## 12. 학습-추론 일치 & 불균형 완화

### 12-1. 해상도/타일 일치

추론을 타일로 한다면(§5) **학습도 같은 타일 크기로** 해야 합니다.

- View crop을 1024 타일로 잘라 **OBB 라벨 좌표를 변환**해 학습
- train과 infer의 입력 분포가 어긋나면 성능이 무너짐

### 12-2. Roughness 152개 불균형 — 동시 공략

타일링과 copy-paste는 해상도뿐 아니라 **불균형도 함께 완화**합니다.

- **타일링** → 작은 객체 비율↑ + Roughness가 여러 타일에서 더 자주 등장
- **Roughness 패치 copy-paste**(§8-4) → 희소 클래스를 인위적으로 증식
- 보조: focal loss / 클래스 가중치 / 희소 클래스 oversampling

> 즉 Roughness 문제는 **해상도(§5) + 불균형(§8-4, §12-2)** 을 함께 건드려야 풀립니다.

## 13. 진단 & ablation

"성능이 안 좋다"의 원인을 구체화하지 않으면 엉뚱한 처방을 하게 됩니다.

### 13-1. 원인 분리해서 보기
- **클래스별 AP** + **AP_small** + **Roughness AP**를 따로 추적
- 오류 유형 분리: 미검출(recall) vs 오검출(precision) vs 위치 부정확(localization)
- 단계 분리: 1단계(View) 누락인지 2단계(주석) 누락인지
- **스캔 품질(깨끗/중간/열화)별**로도 나눠 보기 — 보통 열화 도면에 미검출 집중

### 13-2. 영향 큰 순서대로 ablation
1. **DPI** {300, 400} — 작은 기호 정보량 직접 좌우
2. **Stage 2 imgsz / 타일 크기** {640 통짜 ↔ 1024 타일 ↔ 1280} — Roughness AP가 크게 갈림
3. **전처리: 이진화 유무** — 보통 뺀 쪽이 작은 기호에 유리
4. **증강 강도** — 한 종류씩 올려가며

## 14. 자주 하는 실수 (체크리스트)

- [ ] **소스를 먼저 확인했는가?** → 1-bit/grayscale에 따라 전처리 전략이 달라짐 (§2)
- [ ] **View crop을 통짜로 640에 욱여넣었는가?** → 작은 기호 소멸. 타일링으로 (§5)
- [ ] **전역 deskew로 도면을 틀어버렸는가?** → 테두리 기반 또는 생략+증강 (§6)
- [ ] **이진화를 굳이 추가했는가?** → 회색조가 보통 유리 (§6)
- [ ] **전처리와 증강이 따로 노는가?** → 학습·추론·증강에 일관 적용
- [ ] **검증/테스트 셋에 증강을 적용했는가?** → 증강은 **학습 셋에만**
- [ ] **라벨 동기화를 확인했는가?** → 기하 변형 후 bbox 시각화 검수 (§9)
- [ ] **과하게 열화시켰는가?** → 실제 스캔본보다 심하면 도메인이 멀어짐 (§8)
- [ ] **Mosaic가 소형 객체를 더 작게 만들었는가?** → 밀집 기호엔 약하게/후반 비활성화
- [ ] **학습-추론 해상도가 일치하는가?** → 타일로 추론하면 타일로 학습 (§12)

## 15. 시작 설정값 한눈에 (요약)

```
[소스]    pdfimages -list 로 1-bit/grayscale·DPI 확인  ← 가장 먼저
[추출]    pdfimages -all 로 원본 그대로 (재래스터화 지양)
[DPI]     300 하한 / 작은 기호 많으면 400

[Stage 1] 전체 도면 → imgsz 1024 (밀집 시 1280), 타일 없음

[Stage 2] View crop:
            - crop ≤ 1280px  → imgsz 1280, 타일 없음
            - crop 큼 / 밀집 → 1024 타일 + 20% 오버랩, rotated NMS
          학습도 동일 타일 크기로
          목표: Roughness 기호가 입력에서 ≥ 32px

[전처리]  회색조 + CLAHE (이진화·강한 denoise 생략)
          deskew는 큰 기울임만 / 작으면 증강으로

[증강]    약한 기하 + 강한 스캔 열화 + 실제 패치 copy-paste
          순서: 기하 → 열화 → 압축
          bbox 동기화 시각화 검수 필수

[불균형]  타일링 + Roughness copy-paste (+ focal/가중치)

[진단]    클래스별 AP·AP_small·Roughness AP 따로 추적
          ablation 우선순위: DPI → Stage2 해상도 → 이진화 유무 → 증강 강도
```

### 핵심 우선순위

| 순위 | 항목 | 효과 |
|------|------|------|
| 1 | **소스 점검 + 전처리 최소화** (해상도 정규화 + 회색조, 이진화 제외) | 정보 손실 없이 분포 정리 |
| 2 | **Stage 2 타일링** (통짜 리사이즈 금지) | 소형 기호 누락 즉시 개선 |
| 3 | **실제 스캔 패치 copy-paste** | 라벨 비용 없이 분포·불균형 동시 해결 |
| 4 | **스캔 열화 증강** (압축·morphological) | 스캔본 특유 도메인 갭 메우기 |

> **한 줄 요약**
> 스캔 도면은 대개 **이미 1-bit 흑백으로 들어오므로 소스 점검이 먼저**이고,
> **Stage 2는 절대 통짜 리사이즈로 줄이지 말고 1024 타일로 픽셀을 보존**하며,
> 증강은 **약한 기하 + 강한 스캔 열화 + 실제 패치 copy-paste**가 무게중심이다.
> 이것이 논문의 약점(Roughness 0.54)을 직접 건드리는 출발점이다.
